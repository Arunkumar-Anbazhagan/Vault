# Comprehensive Technical Notes: 

## 1. Introduction to OpenTelemetry Logs
Historically, logs have been the most widely used and legacy-heavy telemetry signal. Unlike metrics (which are numeric representations of system states) and traces (which map the lifecycle of a request), a log is an immutable, append-only data structure that records events occurring in a system at a specific point in time. 

The primary challenge [OpenTelemetry (OTel)](https://opentelemetry.io/docs/what-is-opentelemetry/) solves is the fragmented nature of legacy logging. System logs (e.g., Syslog), infrastructure logs (e.g., Kubernetes events), and application logs often lack standardized formatting and trace context, making them difficult to correlate in a distributed system. OpenTelemetry addresses this by providing a unified [Logs Data Model](https://opentelemetry.io/docs/specs/otel/logs/data-model/) that standardizes how logs are captured, processed, and routed.

## 2. The OpenTelemetry Logs Data Model
The [OTel Logs Data Model](https://github.com/open-telemetry/opentelemetry-specification/blob/main/specification/logs/data-model.md) defines a universal structure for events and log records. This allows existing formats to be mapped into OTel unambiguously, preserving semantics regardless of whether they originate from first-party code, third-party software (like Apache), or the underlying operating system.

### 2.1 Core Top-Level Fields
Every log record in OTel encapsulates two primary kinds of fields: named top-level fields (mandatory/highly common) and attribute fields (arbitrary metadata). 

The official top-level fields include:
* **Timestamp:** The precise time the event occurred, mapped to the origin clock.
* **ObservedTimestamp:** The time when the OpenTelemetry SDK/Collector observed the event.
* **TraceId & SpanId:** The 16-byte request trace identifier and 8-byte span identifier, critical for cross-signal correlation.
* **TraceFlags:** A W3C trace flag (bitmap) indicating options, such as whether a trace is sampled.
* **SeverityText & SeverityNumber:** Standardized indicators of log level (e.g., "ERROR", "INFO"). The numerical mapping ensures consistency across diverse logging backends.
* **Body:** The actual payload or text message of the log record.
* **Resource:** The physical/logical entity that generated the log (e.g., host environment, container).
* **InstrumentationScope:** The library, package, or scope that emitted the log.

### 2.2 Structured vs. Unstructured Logs
According to [OTel Logging Concepts](https://opentelemetry.io/docs/concepts/signals/logs/), logs fall into three categories:
1.  **Unstructured Logs:** Free-form text. While highly human-readable, they require expensive regular expressions to parse in production.
2.  **Semistructured Logs:** Data encoded in formats like JSON, but lacking a guaranteed schema.
3.  **Structured Logs (Recommended):** Logs boasting a stable, consistent schema (well-defined field names and data types) that downstream observability pipelines can reliably process. OpenTelemetry strongly advocates converting unstructured system logs into structured OTel LogRecords.

## 3. Logs Architecture & Bridging
Rather than forcing developers to abandon their favorite logging libraries (e.g., Log4j in Java, Zap in Go), OpenTelemetry is designed to act as a bridge.

### 3.1 The Logs API Components
The [Logs API](https://github.com/open-telemetry/opentelemetry-specification/blob/main/specification/logs/api.md) is primarily built for logging library authors, creating an "appender" or "bridge" to output logs into the OTel pipeline. 

* **LoggerProvider:** The central entry point. Its lifecycle generally matches the application, and it holds configurations for resources and exporters.
* **Logger:** Accessed via the `LoggerProvider`, it generates and emits the `LogRecord` instances. 
* **Log Record Exporter:** Transmits the log records to a consumer, typically standard out (for debugging) or the OpenTelemetry Collector.

### 3.2 Context Propagation & Correlation
One of the most powerful features of OTel is [Context Propagation](https://opentelemetry.io/docs/specs/otel/logs/). When an application is instrumented, OpenTelemetry automatically injects the active `TraceId` and `SpanId` into the existing log bodies. 

This enables tri-dimensional correlation:
1.  **By Time:** Correlating logs, traces, and metrics occurring within the same execution window.
2.  **By Execution Context:** Linking logs to specific traces, enabling engineers to see exactly which log lines belong to a specific failing user request.
3.  **By Resource Context:** Linking application logs to underlying infrastructure metrics using shared Resource attributes.

## 4. Code Integration & The OpenTelemetry Collector
To centralize and process logs, you use the [OpenTelemetry Collector](https://opentelemetry.io/docs/what-is-opentelemetry/), a proxy that receives, processes, and exports telemetry data.

### 4.1 Collector Configuration (YAML)
The Collector utilizes receivers (like the `filelog` receiver to tail text files), processors (to modify or enrich logs), and exporters (to send logs via OTLP to backends like Jaeger, Splunk, or Loki).

Below is an official-compliant example of an OTel Collector configuration designed to ingest raw logs, parse them into a structured format, and export them:

```yaml
receivers:
  filelog:
    include: [ "/var/log/app/*.log" ]
    start_at: end
    operators:
      # Parse unstructured JSON logs into structured attributes
      - type: json_parser
        timestamp:
          parse_from: attributes.time
          layout: '%Y-%m-%dT%H:%M:%S.%LZ'
      # Extract severity
      - type: severity_parser
        parse_from: attributes.level

processors:
  batch:
    send_batch_size: 10000
    timeout: 1s
  attributes:
    actions:
      # Enrich logs with the environment context
      - key: environment
        value: "production"
        action: insert

exporters:
  otlp/logs:
    endpoint: "otel-backend.example.com:4317"
    tls:
      insecure: false

service:
  pipelines:
    logs:
      receivers: [filelog]
      processors: [attributes, batch]
      exporters: [otlp/logs]
