## 1. High-Level Architecture of OpenTelemetry Signals

OpenTelemetry organizes its data gathering capabilities into distinct pillars called **signals**. The primary signals natively supported by the framework are:
* **Tracing:** Tracking the lifecycle and path of a request as it flows through a distributed system.
* **Metrics:** Aggregating numerical data measurements over intervals of time (e.g., CPU usage, request rates).
* **Logging:** Capturing structured or unstructured discrete text events with timestamps.

### Standalone Component Architecture
Every signal is developed and treated as an independent, standalone component within the architecture. However, OpenTelemetry natively provides underlying mechanisms to correlate and connect these distinct data streams to one another (e.g., linking a log entry directly to the trace ID of the transaction that triggered it).

## 2. The Language-Agnostic Specification

At the absolute core of the project lies the [OpenTelemetry Specification](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1). This specification is deliberately language-agnostic to ensure complete consistency, standardization, and interoperability across the entire ecosystem, regardless of whether an application is written in Go, Java, Python, or JavaScript.

While end-users writing application business logic rarely interact with the raw specification documents directly, it serves as the mandatory structural blueprint for all OpenTelemetry implementations.

---

## 3. The Three Pillars of the OpenTelemetry Specification

The formal specification is systematically divided into three major architectural parts:

### A. Terminology and Shared Definitions
* Establishes an official, common vocabulary across all languages and platforms.
* Provides [definitions of terms](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1) to prevent semantic confusion among maintainers, vendor backends, and end-users.

### B. The API Specification
The API (Application Programming Interface) layer defines the conceptual, abstract programming interfaces that developers use to instrument their applications.
* **Implementation Requirements:** It mandates the exact interface definitions that language-specific implementations must strict adherence to.
* **Interoperability:** Ensures that different implementations remain universally compatible with one another.
* **Telemetry Functions:** Defines the core code methods utilized by developers to generate telemetry data.
* *Direct Components:* [Tracing API](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1), [Metrics API](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1), and [OpenTelemetry Logging API](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1).

### C. The SDK Specification
The SDK (Software Development Kit) layer acts as the practical implementation blueprint for the API.
* **Developer Guardrails:** Serves as a technical guide outlining the requirements a language-specific implementation must fulfill to be deemed officially compliant with the API.
* **Data Processing Logic:** Dictates how telemetry data should be configured, batched, sampled, processed, and successfully exported out of the application space.
* *Direct Components:* [Tracing SDK](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1), [Metrics SDK](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1), and [Logs SDK](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1).

## 4. Telemetry Data Standardization & Conventions

Beyond foundational signal architecture, the specification standardizes how the raw data itself is structured:

* **Semantic Conventions:** OpenTelemetry enforces [semantic conventions](https://trainingportal.linuxfoundation.org/learn/course/getting-started-with-opentelemetry-lfs148/overview-of-the-opentelemetry-framework/opentelemetry-framework?page=1) which establish strict, uniform naming rules and metadata interpretations across different technical environments (e.g., standardizing `http.status_code` instead of varying formats like `http_status` or `response_code`). This completely eliminates the need for expensive data normalization on the backend.
* **OpenTelemetry Protocol (OTLP):** The standardized transport protocol designed specifically for encoding and shipping data efficiently from the SDK to collectors or backends.
