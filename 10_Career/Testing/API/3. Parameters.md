## Overview
Parameters are specific pieces of information sent to an API to customize requests, filter results, or provide necessary input. They are categorized based on where they appear in the HTTP request.

---

## 1. Types of Parameters

### **Path Parameters**
* **Location:** Embedded directly in the URL path, usually wrapped in curly braces (e.g., `{id}`).
* **Purpose:** Used to identify a specific resource.
* **Requirement:** These are **required** values.
* **Example:** `GET /pets/{id}` becomes `GET /pets/pet_123`.

### **Query Parameters**
* **Location:** Added after a `?` in the URL. Multiple parameters are joined by `&`.
* **Purpose:** Used for filtering, pagination, and sorting.
* **Requirement:** Usually **optional**.
* **Example:** `/pets?status=AVAILABLE&limit=10`.

### **Header Parameters**
* **Location:** Included within the HTTP headers.
* **Purpose:** Carry metadata such as authentication tokens (`Authorization`), content types (`Content-Type`), or custom tracking IDs.
* **Example:** `Authorization: Bearer <token>`.

### **Body Parameters**
* **Location:** Sent inside the request body.
* **Purpose:** Used for methods that create or update data (**POST**, **PUT**, **PATCH**).
* **Format:** Typically structured as **JSON**, but can be XML or form data.
* **Example:** A JSON object containing details like `name`, `species`, and `age` for a new entry.

---

## 2. Practical Tips for API Testing
* **Check Requirements:** Always verify which parameters are mandatory in the API specification to avoid `400 Bad Request` errors.
* **URL Encoding:** Ensure query parameters with spaces or special characters are properly encoded.
* **Tooling:** Use tools like [Apidog](https://apidog.com/) or Postman to visualize where parameters are placed and to generate the final request URL automatically.
