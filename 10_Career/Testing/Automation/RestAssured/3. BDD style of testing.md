Rest assured uses the BDD style of testing which uses the following 
1. Given()
2. When()
3. Then()
📘 RestAssured DSL Notes

4. given()
   - Defines request setup (preconditions).
   - Common usage:
     • Base URI / Base Path
     • Headers
     • Query parameters
     • Authentication
     • Request body (for POST/PUT)
   - ```java
     Example:
     given().baseUri("https://api.example.com")
            .header("Content-Type", "application/json")
            .body("{\"name\":\"doggo\"}");
        ```

2. when()
   - Specifies the action (HTTP method + endpoint).
   - Common usage:
     • .get("/pets/1")
     • .post("/pets")
     • .put("/pets/1")
     • .delete("/pets/1")
   -```
    Example:
     when().get("/pets/1");

3. then()
   - Defines expected outcome (assertions).
   - Common usage:
     • Status code validation
     • Response body checks
     • Header checks
     • Extracting values
   - Example:
     then().statusCode(200)
           .body("name", equalTo("doggo"));

✅ Why this matters:
- Readable: Mirrors natural language (Given–When–Then).
- Maintainable: Easy to extend with more assertions.
- BDD-aligned: Fits into Cucumber or other BDD frameworks.
- Scalable: Works for simple smoke tests and complex validations.

⚠️ Common pitfalls:
- Forgetting Content-Type for POST/PUT requests.
- Using when() without chaining given() → request may lack setup.
- Overloading then() with too many assertions → harder to debug.
