## 1. Smoke Tests

> **OCR:** Quick tests that verify basic functionality works.

- **Usage:** Running a small set of "critical path" tests (e.g., Can a user log in? Does the homepage load?).
    
- **When to perform:** Immediately after a new build is deployed to a QA or staging environment.
    
- **Why it matters:** It acts as a gatekeeper. If the smoke test fails, the build is "broken," and there is no point in wasting time on deeper testing.
    
- **Why not other tests:** You wouldn't run a full Regression suite here because it takes too long. You need a "Yes/No" answer on the build's health in minutes, not hours.
    

## 2. Regression Tests

> **OCR:** Ensure previously working features haven't broken.

- **Usage:** Re-running existing test cases against a modified application to ensure new code hasn't caused "side effects."
    
- **When to perform:** After bug fixes, feature updates, or configuration changes.
    
- **Why it matters:** In complex systems, fixing a bug in "Module A" often accidentally breaks "Module B." Regression testing catches these "regressions."
    
- **Why not other tests:** Smoke tests are too shallow to catch subtle side-effect bugs, and new feature testing only looks at what's new, ignoring what was already there.
    

## 3. Performance Tests

> **OCR:** Measure how fast your application responds under load.

- **Usage:** Using tools (like JMeter or K6) to simulate 1,000 users hitting the "Checkout" button simultaneously.
    
- **When to perform:** Before a major release or expected traffic spike (e.g., Black Friday).
    
- **Why it matters:** A functional app is useless if it takes 30 seconds to load. This identifies bottlenecks, memory leaks, and server crashes.
    
- **Why not other tests:** Functional tests (Smoke/Regression) only check _if_ it works, not _how fast_ or _how many_ users it can handle.
    

## 4. Security Tests

> **OCR:** Check for vulnerabilities like SQL injection or XSS.

- **Usage:** Scanning for "holes" where hackers could steal data or inject malicious scripts (Cross-Site Scripting).
    
- **When to perform:** Regularly throughout the lifecycle and during dedicated security audits.
    
- **Why it matters:** To prevent data breaches, legal consequences, and loss of user trust.
    
- **Why not other tests:** You can have a perfectly "functional" app that is also incredibly insecure. Normal tests don't try to "break" the app the way a hacker would.
    

## 5. Accessibility Tests (A11y)

> **OCR:** Verify your application works for users with disabilities.

- **Usage:** Checking for screen reader compatibility, keyboard-only navigation, and color contrast.
    
- **When to perform:** During the UI/UX design phase and again once the front-end is built.
    
- **Why it matters:** It’s a matter of inclusivity and, in many regions, the law (WCAG standards). It ensures everyone, including those with visual or motor impairments, can use your product.
    
- **Why not other tests:** Automated functional tests (like Playwright) often ignore whether a button has a "label" for a screen reader unless specifically told to check for it.

## Comparison Summary

|**Test Type**|**Primary Question**|**Focus**|
|---|---|---|
|**Smoke**|Is the build stable?|Stability|
|**Regression**|Did we break anything old?|Reliability|
|**Performance**|Is it fast enough?|Speed/Scale|
|**Security**|Is the data safe?|Vulnerability|
|**Accessibility**|Can everyone use it?|Inclusivity|
