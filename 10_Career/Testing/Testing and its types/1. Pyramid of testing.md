## The Core Strategy: The Testing Pyramid

The Testing Pyramid is a framework that guides how many tests of each type you should write to maintain a fast and reliable suite.

1. **[[1.1 Unit Tests]] (Base):** Tests individual functions or components in isolation. They are fast and cheap.We should have more unit tests
    
2. [[1.2 Integration Tests]] (Middle):** Tests how different modules or services work together (e.g., API talking to a Database). We should have some integration tests
    
3. [[1.3 E2E Tests]] **End-to-End (E2E) Tests (Top):** Tests the entire user journey from start to finish in a real browser. These are the most "expensive" in terms of time and maintenance. Fewer E2E Tests as this tests are prone to breaking as it is brittle in nature
    

> **Testing as "Insurance":** Think of tests as an insurance policy for your code. You pay a small "premium" (time spent writing tests) to avoid a catastrophic "payout" (system failure).
