`adb logcat` streams Android system and app logs; you control _what_ you see with options, filters, and log levels.

---

## 1. Core adb logcat syntax and flags

Basic forms:

- All logs (default buffer, live):
    
    bash
    
    `adb logcat`
    
    Streams logs until you stop with Ctrl+C.
    
- Clear log buffer:
    
    bash
    
    `adb logcat -c`
    
    `-c` clears all log buffers and exits.
    
- Dump and exit (no live tail):
    
    bash
    
    `adb logcat -d`
    
    `-d` prints current logs then exits.
    
- Save to file (PC side):
    
    bash
    
    `adb logcat -v threadtime > logcat.txt`
    
    `-v threadtime` changes output format; redirection writes to `logcat.txt` on your machine.
    

Key options (most useful in day‑to‑day debugging):

|Option / Flag|Example|Usage|
|---|---|---|
|`-c`|`adb logcat -c`|Clear all log buffers, then exit.|
|`-d`|`adb logcat -d`|Dump log and exit (no live streaming).|
|`-f <file>`|`adb logcat -f /sdcard/app.log`|Write logs to a file on the device.|
|`-g`|`adb logcat -g`|Show buffer size and usage, then exit.[](https://adbshell.com/commands/adb-logcat)|
|`-B`|`adb logcat -B`|Binary output (for tools, not humans).[](https://stuff.mit.edu/afs/sipb/project/android/docs/tools/help/logcat.html)|
|`-b <buffer>`|`adb logcat -b main -b system`|Choose buffers: `main`, `system`, `events`, `radio`, `crash`, etc.|
|`-v <format>`|`adb logcat -v threadtime`|Output format (brief, tag, process, time, threadtime, long, raw, etc.).|
|`-n <count>`|`adb logcat -f /sdcard/log.txt -n 5 -r 1024`|Max rotated files when rotating logs (requires `-r`).[](https://stuff.mit.edu/afs/sipb/project/android/docs/tools/help/logcat.html)|
|`-r <kbytes>`|`adb logcat -f /sdcard/log.txt -r 1024`|Rotate log file every `<kbytes>` of output (requires `-f`).|
|`-s`|`adb logcat -s MyTag`|Set default filter to “silent”; only the tags you specify are shown.|
|`--pid <pid>`|`adb logcat --pid 12345`|Show logs only from a specific process ID (Android 7.0+).|
|Filter spec|`adb logcat ActivityManager:I *:S`|Filter by tag and minimum level (details below).[](https://developer.android.com/tools/logcat)|

## 2. Log levels: what, when, and when not
refer more [[2.1 log levels of adb logcat]]

These are ordered from lowest to highest priority:

- `V` / `VERBOSE`: very noisy, internal details.
    
    - Use: deep debugging, tracing flows, temporary detailed prints.
        
    - Avoid in production logs; filter out normally.
        
- `D` / `DEBUG`: debug info useful during development.
    
    - Use: state changes, parameters, non‑critical failures.
        
    - Avoid spamming; in production you typically filter at `INFO` or higher.
        
- `I` / `INFO`: high‑level app state, lifecycle, important but non‑error events.
    
    - Use: user‑visible events, lifecycle transitions, configuration changes.
        
    - Good default viewing level when you want a summary of behavior.
        
- `W` / `WARN`: something unexpected but app can continue.
    
    - Use: recoverable failures, deprecated use, potential problems.
        
    - Don’t overuse; only when you want to be alerted.
        
- `E` / `ERROR`: serious problems where part of the operation failed.
    
    - Use: operation failures, exceptions you catch but need to investigate.
        
    - Look here first when debugging crashes or functional bugs.
        
- `F` / `ASSERT` / `WTF`: should never happen; system or app invariants broken.
    
    - Use: truly impossible states; usually you’d crash or abort after.
        
    - Very rare; treat them like urgent bugs.
        
- `S` / `SILENT`: special; used only in filter specs to mean “print nothing” for that tag.[](https://developer.android.com/tools/logcat)
    

Filtering by level in `logcat`:

bash

`# Only warnings and above from all tags adb logcat *:W # Only errors (and fatal) from your app’s tag adb logcat MyAppTag:E *:S`

In both cases, messages with the specified level **and higher** are shown.

## 3. Filter specs: tags + levels

Filter spec syntax:[](https://developer.android.com/tools/logcat)

bash

`adb logcat <tag1>:<level1> <tag2>:<level2> ... *:<defaultLevel>`

Examples:

- Only your app tag, info and above:
    
    bash
    
    `adb logcat MyAppTag:I *:S`
    
    Shows `MyAppTag` at `INFO`, `WARN`, `ERROR`, `FATAL`; everything else is silent.[](https://developer.android.com/tools/logcat)
    
- ActivityManager INFO+ and your tag DEBUG+, nothing else:
    
    bash
    
    `adb logcat ActivityManager:I MyAppTag:D *:S`
    
    Good when debugging interactions with system components.[](https://developer.android.com/tools/logcat)
    

If you omit `*:S`, you’ll see other tags at their default level too.[](https://developer.android.com/tools/logcat)

## 4. Single‑command patterns: get PID and feed into logcat

You asked specifically for **single commands** that obtain the PID and pass it to `adb logcat` with verbose or other flags. Common patterns:

## 4.1. Basic: logcat for app PID (shell side)

Inside `adb shell`:

bash

`adb shell "logcat --pid=$(pidof -s com.example.myapp)"`

- `pidof -s <package>` returns the process ID of your app.[](https://stackoverflow.com/questions/32735735/adb-shell-logcat-with-package-name)
    
- `logcat --pid=<pid>` restricts output to that process.
    

With verbose level and your tag only:

bash

`adb shell "logcat --pid=$(pidof -s com.example.myapp) MyTag:V *:S"`

You get only logs from that PID, for tag `MyTag`, at `VERBOSE` and above.

## 4.2. From your dev machine (no interactive shell)

Common pattern (works in many shells; adjust quoting for PowerShell):

bash

`adb logcat --pid=$(adb shell pidof -s com.example.myapp) MyTag:V *:S`

- First `adb shell pidof -s ...` resolves the PID on the device.
    
- That PID is then passed to `adb logcat --pid=<pid>` in the same command line.
    

For **all tags** from that process, with detailed timestamps:

bash

`adb logcat --pid=$(adb shell pidof -s com.example.myapp) -v threadtime`

## 4.3. Multiple PIDs (e.g., app + service process)

If your app spawns another process (e.g. `com.example.myapp:service`):

bash

`adb logcat \   --pid=$(adb shell pidof -s com.example.myapp) \  --pid=$(adb shell pidof -s com.example.myapp:service) \  MyTag:D *:S`

You now see only your chosen tag from both processes, at `DEBUG` and above.[](https://whhone.com/posts/adb-logcat/)

## 5. Practical scenarios (when to use what)

## Scenario A: App crashes or throws exceptions

- Command:
    
    bash
    
    `adb logcat *:E`
    
- Reason: focus on `ERROR` and `FATAL` from the whole system, quickest way to see stack traces.
    
- Optionally narrow by PID once you know the process:
    
    bash
    
    `adb logcat --pid=$(adb shell pidof -s com.example.myapp) *:E`
    

## Scenario B: Debugging your own app’s flow

- Command:
    
    bash
    
    `adb logcat --pid=$(adb shell pidof -s com.example.myapp) MyAppTag:D *:S`
    
- Reason: see your tag’s debug logs plus info/warn/error for that process only; avoids system noise.
    
- Use `D` when iterating on logic; bump most logs down to `V` and filter them out when you move closer to production.
    

## Scenario C: Telephony/radio or system behavior

- Command:
    
    bash
    
    `adb logcat -b radio -v time`
    
- Reason: radio buffer (network/telephony) is separate; using `-b radio` focuses on those logs.
    
- You can combine: `adb logcat -b radio -b main *:I` to see both radio and main info+.
    

## Scenario D: Long‑running capture to file

- Command:
    
    bash
    
    `adb logcat -v threadtime -f /sdcard/app.log -r 2048 -n 5`
    
- Reason: logs with timestamps and rotating files (5 files, 2 MB each) on the device.
    
- Later you can `adb pull /sdcard/app.log` and friends.
