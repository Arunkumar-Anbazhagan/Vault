## microsoft/playwrightPowered by Gemini

Playwright is a web automation and testing framework that supports Chromium, Firefox, and WebKit browsers using a unified API. It provides capabilities for end-to-end testing, scripting, and integration with AI agents.

The framework's functionality is primarily driven by the [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) package, which delivers browser-agnostic automation and manages browser installations and environment setup. Playwright employs a client-server architecture for remote interaction and control of browser instances. This architecture manages object lifecycles and dispatches messages across potentially separate processes, as detailed in [Client-Server Communication Architecture](https://codewiki.google/github.com/microsoft/playwright#playwright-core-functionality-and-architecture-client-server-communication-architecture).

Playwright achieves cross-browser compatibility through specific implementations, such as the Firefox Juggler backend for automation and custom WebKit embedders for macOS and Windows. These components facilitate deep interaction with browser internals, as outlined in [Browser Management and Patching](https://codewiki.google/github.com/microsoft/playwright#browser-management-and-patching). The framework also offers advanced control over browsers and devices, including Android device automation and Electron application control.

Various tools and utilities enhance Playwright's development and analysis workflows. These include an interactive dashboard for remote browser control, an HTML reporter for test results, and a recorder for code generation and inspection. Post-mortem analysis of automation runs is supported by the trace viewer. These extensions are part of the broader [Playwright Ecosystem and Extensions](https://codewiki.google/github.com/microsoft/playwright#playwright-ecosystem-and-extensions). Additionally, Playwright provides component testing functionalities for UI components within a browser environment.

Development and operations follow established workflows for API extensions, internal tool creation, and dependency management. These processes also cover continuous integration failure analysis and reporting, which contribute to project stability and quality. For more details, see [Development and DevOps Workflows](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows).

### Monorepo Structure and Development Workflow

The Playwright project employs a monorepo structure, organizing its codebase into distinct packages to manage various functionalities and facilitate development. This structure is detailed in the developer documentation located at [`CLAUDE.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md).

**Package Organization:** The monorepo categorizes packages based on their roles:

- **Monorepo Packages:** These include core components such as [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) (the browser automation engine), [`playwright`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/cli.js#L20) (the public package combining the test runner and automation libraries), [`@playwright/test`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L36) (the entry point for the test runner), [`@playwright/client`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L8) (a standalone client), and [`protocol`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/webSocketMock.ts#L128) (defining internal Remote Procedure Call, RPC, communications). The [`protocol`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/webSocketMock.ts#L128) package generates type definitions ([`channels.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L9)) from its specification ([`protocol.yml`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L9)) to ensure consistent communication.
- **Browser Packages:** Dedicated packages for each supported browser (Chromium, Firefox, WebKit) and their respective binary distributions (e.g., [`packages/playwright-chromium`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-chromium), [`packages/playwright-browser-chromium`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-browser-chromium)).
- **Tooling Packages:** These encompass developer tools like [`packages/html-reporter`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/html-reporter), [`packages/trace-viewer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer), [`packages/recorder`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder), [`packages/web`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/web) (providing shared UI components), and [`packages/injected`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected) (scripts injected into browser pages).
- **Component Testing Packages:** Experimental packages, including [`packages/playwright-ct-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core), [`packages/playwright-ct-react`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-react), and [`packages/playwright-ct-vue`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-vue), extend the testing framework to support component-level testing for popular frontend frameworks.

- The [`CLAUDE.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md) document outlines that the [`tests/`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L34) directory contains all test suites, covering library features, test runner functionality, Multi-client Protocol (MCP) tools, and component tests.

**Development Workflow:** The development workflow is designed to ensure code quality and consistency. A full build can be initiated with [`npm run build`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L40), while [`npm run watch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L3) is recommended for local development, as it automatically regenerates necessary files like types, channels, and validators.

Linting and type checking are critical steps, enforced by [`npm run flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L3). This command executes a suite of checks, including ESLint, TypeScript compilation, documentation linting, dependency verification, channel generation, type generation, test linting, and code snippet validation. Developers are required to run [`flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L52) before committing any changes.

**Internal Dependency Management (DEPS System):** The project utilizes a strict [`DEPS`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L90) system to manage internal import boundaries, enforced by [`npm run flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L3). This system, configured through [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files, prevents unintended dependencies and ensures a clean architecture. A fundamental rule is that client code cannot directly import server code, and vice versa; all communication between these layers must adhere to the defined protocol. Files marked as [`"strict"`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/types/types.d.ts#L24047) in [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) can only import explicitly declared dependencies. This rigorous approach helps maintain modularity and prevent circular dependencies. The core architecture of Playwright's client-server communication is further elaborated in [Client-Server Architecture and Dependency Management](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-client-server-architecture-and-dependency-management).

**Coding and Commit Conventions:** The project follows specific coding conventions for exported classes to clearly distinguish between private, internal, and public methods. A semantic commit message convention, [`label(scope): description`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L110), is used, with labels such as [`fix`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L108), [`feat`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L112), [`chore`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L112), [`docs`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/cli.js#L25), [`test`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/mcp/fixtures.ts#L86), and [`devops`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L112). This practice ensures commit history is clear and descriptive. Branch naming for bug fixes follows the [`fix-<issue-number>`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L140) format. Maintaining a clear review trail is important, thus developers are instructed to avoid amending commits or force-pushing without explicit guidance. ### Reporting Issues and Security Vulnerabilities

| Feature | General Bug Report | Security Vulnerability Report |
| --- | --- | --- |
| **Reporting Channel** | GitHub Issues (using Bug Report template) | Microsoft Security Response Center (MSRC) portal or email |
| **Confidentiality** | Public | Confidential |
| **Key Information** | Repro steps, expected vs. actual, system info (`envinfo`) | Type of issue, source file paths, configuration, impact |
| **Reproduction** | Minimal, isolated repro code | Step-by-step instructions, proof-of-concept/exploit code (if possible) |
| **Why it Matters** | Speeds up verification and resolution | Ensures coordinated disclosure and swift remediation |
| **Dependencies** | Avoid extra libraries unless necessary | Not explicitly mentioned, but assumed similar best practices |

When reporting bugs within the Playwright project, contributors should adhere to specific guidelines to ensure efficient resolution. It is crucial to utilize the provided bug report template, offering comprehensive details, clear steps to reproduce the issue, and a comparison of expected versus actual outcomes. System information, obtainable via [`npx envinfo --preset playwright`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/FILING_ISSUES.md?plain=1#L3), should also be included. A key requirement for bug reports is the provision of a minimal, isolated reproduction, which involves setting up a fresh project and including only the essential code and DOM elements to demonstrate the bug. This approach helps in debugging and verification by eliminating extraneous factors. Without a reproducible example, issues are often unresolvable, as they may stem from misconfigurations or usage errors rather than core Playwright bugs. For further guidance on creating effective minimal reproductions, external resources are available, alongside Playwright's debugging tools.

For security vulnerabilities, a distinct and formal process is followed, reflecting Microsoft's commitment to security across its GitHub repositories. Security issues should *not* be reported via public GitHub issues. Instead, individuals are directed to report vulnerabilities through the Microsoft Security Response Center (MSRC) via [`https://msrc.microsoft.com/create-report`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/SECURITY.md?plain=1#L13) or [`secure@microsoft.com`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/SECURITY.md?plain=1#L15), with PGP encryption recommended for email submissions. Reports should include critical information such as the type of issue, full paths of affected source files, the location of the vulnerable code, any special configurations, step-by-step reproduction instructions, proof-of-concept code, and the potential impact of the vulnerability. Microsoft aims to respond to reports within 24 hours. The process adheres to the Coordinated Vulnerability Disclosure principle, and detailed, high-quality reports may be eligible for awards under the Microsoft Bug Bounty Program. All communications regarding security issues should preferably be in English.

## Playwright Core Functionality and Architecture

Playwright's core functionality is encapsulated within the [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) package, which provides browser-agnostic automation capabilities without bundling browser binaries. This design allows for a lightweight core that can interact with various browsers by dynamically managing their installation and configuration.

The [`packages/playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core) directory serves as the foundation for Playwright's operations. Its primary entry point, [`packages/playwright-core/index.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/index.js), enforces a minimum Node.js version, ensuring compatibility before the library is loaded. The [`README.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/workspace.js#L148) file within this package clarifies its purpose as a no-browser flavor of Playwright, focusing on core functionality. Third-party software notices and licensing information are maintained in [`packages/playwright-core/ThirdPartyNotices.txt`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/ThirdPartyNotices.txt).

A significant aspect of [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) is its management of browser installations and environment setup. The [`packages/playwright-core/bin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/bin) directory contains scripts that automate the installation and reinstallation of browsers such as Chrome, Edge, and WebKit across different operating systems. ### Client-Server Communication Architecture

Playwright's client-server model facilitates communication between the user's script and the browser automation backend. This architecture is crucial for managing the lifecycle of objects and dispatching messages across potentially separate processes. The client-side interacts with a representation of backend objects through a channel-based system, while the server-side handles the actual browser control.

The foundation for client-side objects communicating with the Playwright backend is the [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32) class, defined in [`packages/playwright-core/src/client/channelOwner.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts). Most client-side Playwright objects, such as [`Browser`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/browser.ts#L70) and [`BrowserContext`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js#L896), inherit from [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32). This class manages the object's lifecycle and its communication with the backend over a designated channel. It provides structured API call wrapping, including validation against a protocol schema, logging, error handling, and stack trace capture for debugging. [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32) also automatically manages event subscriptions on the protocol level, ensuring that the client only requests event notifications from the backend when active listeners are present, thereby optimizing network traffic.

The actual remote communication on the client-side is handled by the [`Connection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/connection.ts#L69) class, found in [`packages/playwright-core/src/client/connection.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/connection.ts). This class is responsible for managing remote objects, sending messages to the server, dispatching incoming messages, and handling errors. It uses object factories to dynamically create [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32) subclasses based on messages received from the server.

On the server-side, communication with the client is primarily facilitated through **Dispatchers**, located within the [`packages/playwright-core/src/server/dispatchers`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/dispatchers) directory. These classes act as proxies for server-side [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49) instances, translating client commands into actions on the browser and sending events back to the client. The [`Dispatcher`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/runner/dispatcher.ts#L35) class handles the lifecycle of server objects, dispatches events to the client, and executes client commands. A [`RootDispatcher`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/dispatchers/dispatcher.ts#L174) serves as the entry point for the object hierarchy. The [`DispatcherConnection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/dispatchers/dispatcher.ts#L192) class manages the communication channel, object registration, garbage collection of stale dispatchers, and serialization/deserialization of messages.

The [`packages/playwright-core/src/protocol`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/protocol) directory defines the structure of messages exchanged between the client and server. The [`packages/playwright-core/src/protocol/serializers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/protocol/serializers.ts) file within this directory handles the conversion between native JavaScript values and a [`SerializedValue`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/versions/traceV8.ts#L32) format for transmission. This includes handling complex objects, circular references, and specific types like [`Date`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/eslint.config.mjs#L237) and [`RegExp`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/network.md?plain=1#L349). The [`packages/playwright-core/src/protocol/validator.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/protocol/validator.ts) and [`packages/playwright-core/src/protocol/validatorPrimitives.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/protocol/validatorPrimitives.ts) files define the Playwright protocol's object schemes and the foundational system for validating data structures against these schemes, ensuring type safety and consistency in message exchange.

This client-server architecture allows Playwright to support both in-process and out-of-process executions, providing flexibility and enabling advanced features like remote browser automation.

### Remote Connection Management and Server Implementations

Playwright manages interactions between clients and browser instances through a remote connection architecture, offering different server implementations and operational modes. The central component for this is the [`PlaywrightConnection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightConnection.ts#L34) class, found in [`packages/playwright-core/src/remote/playwrightConnection.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightConnection.ts), which orchestrates message dispatch, handles connection lifecycle events, and ensures proper resource cleanup. This class utilizes a [`ServerTransport`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/serverTransport.ts#L22) (defined in [`packages/playwright-core/src/remote/serverTransport.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/serverTransport.ts)) to send and receive messages, processing them through a dispatcher connection that routes incoming JSON messages to appropriate handlers and serializes outgoing messages.

Playwright provides three main server implementations:

- **[`PlaywrightServer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightServer.ts#L48)**: This class, located in [`packages/playwright-core/src/remote/playwrightServer.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightServer.ts), serves as the primary WebSocket server. It handles incoming WebSocket connections and orchestrates the launch and management of browser instances or Android devices based on client requests and server configuration. It supports various operational modes:
	- **Reuse Browsers Mode**: This mode attempts to find and reuse an existing browser instance that matches the requested browser name and launch options. If no suitable browser is found, it launches a new one. This is designed to keep browsers alive for subsequent connections, closing idle contexts but not the browser itself.
		- **Connect Mode**: Primarily used for connecting to an existing browser, often in "extension" scenarios. It ensures a browser of the specified type is available, launching one if necessary, and marks it as not reusable by subsequent connections.
		- **Pre-launched Browser Mode**: When a specific browser is pre-launched, this mode ensures that only this particular browser remains open and is used for all incoming connections. A similar mode exists for pre-launched Android devices.
		- **Launch New Browser Mode**: This is the default operational mode where each new client connection triggers the launch of a new browser instance. This mode can also configure a SOCKS proxy for newly launched browsers if specified in the request.
- **[`PlaywrightPipeServer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightPipeServer.ts#L29)**: Defined in [`packages/playwright-core/src/remote/playwrightPipeServer.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightPipeServer.ts), this class creates and manages a named pipe server. It allows multiple clients to connect and interact with a single, pre-launched browser instance, sharing it among them. This is useful for scenarios where a Playwright browser runs as a shared service, making a single browser instance available to multiple clients concurrently.
- **[`PlaywrightWebSocketServer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightWebSocketServer.ts#L26)**: Found in [`packages/playwright-core/src/remote/playwrightWebSocketServer.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/playwrightWebSocketServer.ts), this server provides a specialized WebSocket interface for connecting to an already running browser instance. It is designed to attach to an existing [`Browser`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/browser.ts#L70) instance, allowing clients to control it via WebSockets. It operates in a "pre-launched browser mode," providing a shared browser to connected clients.

These server implementations utilize a common [`ServerTransport`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/serverTransport.ts#L22) interface to abstract the underlying communication protocol. For WebSockets, the [`WebSocketServerTransport`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/serverTransport.ts#L31) directly wraps the WebSocket object. For named pipe connections, the [`SocketServerTransport`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/remote/serverTransport.ts#L58) handles message framing by using null-terminated messages to reconstruct data streams from incoming buffers. This modular design allows Playwright to support diverse remote connection scenarios, ranging from individual browser launches to shared browser instances and pre-configured environments.

## Browser Management and Patching

The Playwright framework integrates and manages various browser engines to facilitate automation and testing. This process involves synchronizing upstream browser files, enabling automation and debugging for Firefox, and building and launching specialized WebKit browsers for different operating systems. These browser-specific functionalities and extensions are critical for Playwright's cross-browser compatibility.

A central script, [`browser_patches/roll_from_upstream.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/roll_from_upstream.sh), maintains consistency by synchronizing browser-related files from a [`playwright-browsers`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/upload_flakiness_dashboard.sh#L27) checkout into Playwright's local browser patch directories. This ensures that the framework always uses the intended versions of Firefox and WebKit components, including patches, configurations, and utilities.

For Firefox, Playwright employs a specialized Juggler backend, located in [`browser_patches/firefox/juggler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler). This backend enables comprehensive control over Firefox's browsing contexts, network activity, JavaScript execution, and screencasting capabilities. It establishes cross-process communication channels and manages browser settings for remote debugging and automation. The specific Firefox version and commit hash are pinned in [`browser_patches/firefox/UPSTREAM_CONFIG.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/UPSTREAM_CONFIG.sh) for stability.

Playwright also supports a patched WebKit browser. The specific Git commit hash for WebKit is managed in [`browser_patches/webkit/UPSTREAM_CONFIG.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/webkit/UPSTREAM_CONFIG.sh). The [`browser_patches/webkit/embedder`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/webkit/embedder) directory contains the build processes for Playwright's custom WebKit embedders, supporting both macOS and Windows. These embedders handle the browser's lifecycle, manage [`WKWebView`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/webkit/embedder/Playwright/mac/AppDelegate.m#L301) instances for both headful and headless operations, configure user data and proxy settings, and facilitate UI interactions, navigation, and JavaScript dialogs. On macOS, Objective-C and CMake are used to build the embedder, managing application lifecycle and [`WKWebView`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/webkit/embedder/Playwright/mac/AppDelegate.m#L301) configurations. On Windows, C++ and CMake are used, focusing on the application entry point, main window management, and handling [`WKPage`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/wkPage.ts#L55) embedding for various browser events. A utility script, [`browser_patches/webkit/pw_run.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/webkit/pw_run.sh), is responsible for launching the Playwright browser on macOS or Linux, configuring necessary environment variables for dynamic libraries and bundles.

Additionally, Playwright includes a utility called [`winldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/installation/npmTest.ts#L35), located in [`browser_patches/winldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd). This tool functions similarly to [`ldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L3) on Linux, allowing developers to list the dynamic link library (DLL) dependencies of Windows executables and libraries. This is useful for diagnosing dependency issues within the Windows environment.

### Firefox Juggler Backend for Automation and Debugging

The Juggler backend for Firefox provides the infrastructure for automating and debugging the Firefox browser, allowing Playwright to control browser behavior, manage network activity, and execute JavaScript. This backend manages interactions across multiple processes within Firefox, including communication channels and event handling.

Communication between different parts of the Juggler system, as well as between the browser process and content processes, is facilitated by a generic, bidirectional message passing mechanism, the [`SimpleChannel`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/SimpleChannel.js#L11) found in [`browser_patches/firefox/juggler/SimpleChannel.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/SimpleChannel.js). This channel supports sending and receiving messages asynchronously, often used for remote procedure calls (RPC). A cross-process communication pipe, implemented in [`browser_patches/firefox/juggler/pipe/nsRemoteDebuggingPipe.h`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/pipe/nsRemoteDebuggingPipe.h) and [`browser_patches/firefox/juggler/pipe/nsRemoteDebuggingPipe.cpp`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/pipe/nsRemoteDebuggingPipe.cpp), ensures robust asynchronous communication for remote debugging by handling message framing and thread management. Essential utility functions, located in [`browser_patches/firefox/juggler/Helper.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/Helper.js), aid in event management and unique ID generation.

Browser management and lifecycle are primarily handled by the [`Juggler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/components/Juggler.js#L49) component, defined in [`browser_patches/firefox/juggler/components/Juggler.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/components/Juggler.js). This component serves as an entry point for initializing the automation environment, setting up remote debugging, and registering [`JSWindowActors`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/components/Juggler.js#L21) for frame-level interactions. It observes critical browser lifecycle events to determine readiness for automation. The [`TargetRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js#L106), located in [`browser_patches/firefox/juggler/TargetRegistry.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js), acts as a central manager for all browser targets (pages) and contexts. Network control within the Juggler backend is managed by the [`NetworkObserver`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/NetworkObserver.js#L600), found in [`browser_patches/firefox/juggler/NetworkObserver.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/NetworkObserver.js). Interaction with content processes and JavaScript execution is managed through several components located in [`browser_patches/firefox/juggler/content`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/content). ### Windows DLL Dependency Utility (winldd)

The [`winldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/installation/npmTest.ts#L35) utility, found in the directory [`browser_patches/winldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd), provides a mechanism for identifying and listing the dynamic link library (DLL) dependencies of Windows executables and DLL files. This functionality is comparable to the [`ldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L3) command available on Linux systems.

The core implementation, detailed in [`browser_patches/winldd/PrintDeps.cpp`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp), parses command-line arguments, loads specified modules, and then extracts import descriptors from the Portable Executable (PE) format of each module. It resolves the full paths of imported DLLs and outputs them to standard output, indicating "not found" if a DLL cannot be located. A key design choice is the use of [`DONT_RESOLVE_DLL_REFERENCES`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L79) during initial [`LoadLibraryEx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L79) calls, which prevents recursive loading and potential side effects, focusing solely on dependency extraction. Static linking of C runtime libraries minimizes the tool's own dependencies, with [`dbghelp.dll`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L6) being the only dynamic dependency.

The build process for [`winldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/installation/npmTest.ts#L35) is orchestrated by [`browser_patches/winldd/build.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/build.sh), which executes a Windows-specific batch script to compile the utility. The output, [`PrintDeps.exe`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L12), is then managed by an archiving script, [`browser_patches/winldd/archive.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/archive.sh), which creates a [`.zip`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock.md?plain=1#L419) archive containing the executable. This archive is designed for distribution and ensures that [`PrintDeps.exe`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L12) is bundled with every [`npm`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L3) package of Playwright. A [`browser_patches/winldd/clean.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/clean.sh) script is available for removing build artifacts.

## Developer Tools and Utilities

The Playwright project includes a variety of tools and utilities designed to support its development, build, and deployment processes. These utilities address tasks ranging from environment setup for mobile automation to sophisticated code generation and quality assurance.

For Android development, Playwright provides scripts to manage the Android SDK and Virtual Devices (AVDs). For instance, [`utils/avd_install.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_install.sh) automates the local installation of the Android SDK and emulator, ensuring a self-contained environment. Building upon this, [`utils/avd_recreate.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_recreate.sh) manages the creation and configuration of AVDs, deleting existing ones and creating new ones with specific profiles and architectures. Once an AVD is ready, [`utils/avd_start.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_start.sh) starts an emulator and waits for it to become fully operational, while [`utils/avd_stop.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_stop.sh) handles the graceful shutdown of running emulators. Furthermore, [`utils/build_android_driver.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build_android_driver.sh) automates the building of Android driver and test APKs, preparing them for deployment or CDN upload.

The project's build system is comprehensive, handling package orchestration and deployment. The [`utils/build`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build) directory contains scripts that manage core and web package builds, integrate with bundlers like [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23) and [`vite`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L159), and handle dynamic imports for vendored packages as defined in [`utils/build/utilsBundleMapping.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/utilsBundleMapping.js). Specific scripts like [`utils/build/build-playwright-driver.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build-playwright-driver.sh) are responsible for packaging platform-specific Playwright driver archives, bundling [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) with a Node.js runtime and protocol definitions. The Docker integration in [`utils/docker`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker) further extends these capabilities, enabling the building, tagging, and publishing of Playwright Docker images for various architectures and Ubuntu versions. This process includes packaging [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) into a tarball for use within Docker builds and attaching End-of-Life manifests.

Code generation is a significant aspect of Playwright's development utilities, ensuring consistency across different components and languages. Scripts like [`utils/generate_channels.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_channels.js) automatically generate TypeScript definitions and validation schemas for Playwright protocol channels, events, and types from a protocol specification. Similarly, [`utils/generate_dotnet_channels.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_dotnet_channels.js) generates C# classes for the Playwright.NET client. Other generators include [`utils/generate_cli_help.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_cli_help.js) for CLI help documentation and [`utils/generate_types`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types) for API TypeScript declaration files, including comprehensive overrides for core Playwright APIs and test APIs. For browser protocol types, [`utils/protocol-types-generator`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/protocol-types-generator) orchestrates the generation of TypeScript [`.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js#L164) files for Chromium, Firefox, and WebKit.

### Android Development Environment Management

Playwright includes utilities for managing the Android development environment, essential for automating mobile browsers and applications. This involves a set of scripts that streamline the installation of the Android SDK, the creation and management of Android Virtual Devices (AVDs), the building of Android driver and test APKs, and the control of emulator lifecycles.

The installation of the Android SDK is handled by the script in [`utils/avd_install.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_install.sh). This script automates the download and setup of necessary components such as command-line tools, the emulator, and specific platform SDKs, ensuring a self-contained local environment that avoids conflicts with existing [`ANDROID_HOME`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_stop.sh#L5) configurations. Once the SDK is in place, the [`utils/avd_recreate.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_recreate.sh) script is used to manage AVDs. It configures environment variables, determines the correct Android architecture, deletes any pre-existing [`android35`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_start.sh#L22) AVDs, installs required SDK components, and then creates a new [`android35`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_start.sh#L22) AVD. To control the emulator's lifecycle, [`utils/avd_start.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_start.sh) initiates an Android emulator by name, ensuring it is fully operational before proceeding with further commands. Conversely, [`utils/avd_stop.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/avd_stop.sh) terminates running emulators by identifying and sending kill signals to active ADB processes.

The process for building Android driver and test APKs is automated by [`utils/build_android_driver.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build_android_driver.sh). This script compiles the necessary APK files and prepares them for distribution, including zipping them into an archive suitable for CDN upload. These utilities collectively enable Playwright's mobile automation capabilities by providing a robust and repeatable way to set up and manage the Android testing environment. For a broader view of Playwright's utility scripts, see [Developer Tools and Utilities](https://codewiki.google/github.com/microsoft/playwright#developer-tools-and-utilities).

### Advanced Build Orchestration and Packaging

Playwright's build system orchestrates the compilation, bundling, and packaging of its various components, encompassing both core functionality and web-based applications. This system supports both one-time builds and continuous watch modes for development, leveraging modern tools such as [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23) and [`vite`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L159).

The central build script, located at [`utils/build/build.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js), defines a series of operations using a step-based approach. These steps can execute external commands, perform TypeScript compilation and bundling via [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23), or run custom callbacks. For web-oriented packages like the HTML reporter, recorder, trace viewer, dashboard, and browser extension UI, the build system integrates [`vite`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L159), indicating a hybrid strategy that optimizes for different types of artifacts.

A key aspect of the build process is its sophisticated handling of module dependencies and imports. The [`dynamicImportToRequirePlugin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L502) within [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23) transforms import statements, particularly for internal isomorphic and utility aliases, and manages vendored third-party packages. This plugin ensures that external dependencies, explicitly mapped in [`utils/build/utilsBundleMapping.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/utilsBundleMapping.js), are correctly resolved from a unified utility bundle rather than separate [`node_modules`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L27) installations. This vendoring strategy contributes to a consistent and optimized runtime environment by bundling critical components like [`coreBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/installation/fixture-scripts/driver-client.js#L18), [`utilsBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js#L101), and [`babelBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L649) separately.

Beyond core compilation, the build system is responsible for creating platform-specific Playwright driver archives. The [`utils/build/build-playwright-driver.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build-playwright-driver.sh) script automates this process by packaging [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1), a specific Node.js runtime, API definitions ([`api.json`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L601)), and protocol specifications ([`protocol.yml`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L9)) into self-contained [`.zip`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock.md?plain=1#L419) archives. This includes managing Node.js versioning, ensuring [`npm install`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L27) skips browser downloads to keep driver packages lean, and resetting file timestamps for consistency.

The system also includes utilities for generating detailed bundle reports, managed by [`utils/build/bundle_report.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/bundle_report.js), which analyze [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23) output metadata to provide insights into inlined file sizes, external dependencies, and third-party license information. This facilitates both bundle size optimization and legal compliance. For release management, the [`utils/build/update_canary_version.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/update_canary_version.js) script dynamically updates version numbers across the workspace, incorporating release prefixes and timestamps.

### Docker Image Building and Publishing

The process of building, tagging, and publishing Playwright Docker images for multiple architectures and Ubuntu versions is managed through scripts located in the [`utils/docker`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker) directory. This system ensures that Playwright is packaged into Docker images and made available with appropriate metadata.

The core of this process involves the [`utils/docker/build.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/build.sh) script, which prepares individual Docker images for specific architectures and Ubuntu distributions. Before a Docker image is built, the [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) package is first bundled into a tarball. This tarball is then included in the Docker build process, allowing Playwright's dependencies and browser downloads to be efficiently handled within the container.

Orchestrating the end-to-end workflow is the [`utils/docker/publish_docker.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/publish_docker.sh) script. This script handles the validation of the Playwright version and release channel, ensuring that only stable versions are published. It builds and pushes multi-architecture images for different Ubuntu versions, specifically [`jammy`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/docker.md?plain=1#L218) (Ubuntu 22.04) and [`noble`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/ci.md?plain=1#L211) (Ubuntu 24.04), utilizing the [`publish_docker_images_with_arch_suffix`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/publish_docker.sh#L64) function. A key aspect of this publication process is the attachment of End-of-Life (EOL) manifests to the images using the [`oras`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/publish_docker.sh#L48) command-line tool, a process managed by the [`attach_eol_manifest`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/publish_docker.sh#L44) function. This adds important lifecycle metadata to the published Docker images. Finally, the script uses the [`publish_docker_manifest`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/publish_docker.sh#L90) function to create and push multi-architecture manifest lists, which allows a single Docker image tag to correctly resolve to the appropriate image for either [`amd64`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/docker/build.sh#L6) or [`arm64`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/intro-js.md?plain=1#L308) architectures. This approach simplifies consumption for users by providing a unified tag for different environments.

### Documentation and Protocol Type Generation

Playwright employs a suite of tools and scripts to automatically generate various forms of documentation and type definitions, ensuring consistency and accuracy across its APIs and browser protocols. This process involves converting structured documentation into TypeScript declaration files, JSON representations, and C# API definitions. It also handles the generation of type definitions for the debugging protocols of Chromium, Firefox, and WebKit directly from their source.

The core TypeScript type generation for Playwright's APIs is orchestrated by the [`utils/generate_types/index.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/index.js) script. This script parses API documentation from several source directories, including those for core, Electron, mobile, test, and test reporter APIs. It uses a [`TypesGenerator`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/index.js#L31) class to transform this parsed information into TypeScript interfaces and types. The process incorporates custom type definitions and overrides defined in files like [`utils/generate_types/overrides.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides.d.ts), [`utils/generate_types/overrides-test.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides-test.d.ts), and [`utils/generate_types/overrides-testReporter.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides-testReporter.d.ts). These override files enhance type safety, provide specific method overloads, and integrate with external libraries and frameworks. For instance, [`utils/generate_types/overrides.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides.d.ts) augments core interfaces like [`Page`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/page.ts#L164) and [`Locator`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/consoleApi.ts#L28) with refined type signatures, while [`utils/generate_types/overrides-test.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides-test.d.ts) defines types for Playwright Test configurations, fixtures, and assertion utilities. The [`utils/generate_types/overrides-testReporter.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/overrides-testReporter.d.ts) file specifically details the structure for test reporting, including overall run results and detailed JSON report schemas. The [`parseOverrides`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/parseOverrides.js#L26) utility, found in [`utils/generate_types/parseOverrides.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_types/parseOverrides.js), processes these TypeScript override files to inject comments and additional code into the generated output.

Beyond TypeScript, Playwright also generates JSON and C# representations of its API documentation. The [`utils/doclint/generateApiJson.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/generateApiJson.js) script generates a structured JSON representation of the API, converting the internal [`Documentation`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/documentation.js#L75) objects into a JSON-compatible format. For browser protocol type definitions, Playwright utilizes the [`utils/protocol-types-generator`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/protocol-types-generator) directory. Its [`index.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/eslint.config.mjs#L24) file orchestrates the generation of TypeScript declaration files ([`protocol.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L959)) for Chromium, Firefox, and WebKit debugging protocols. The Playwright internal protocol, which defines communication between the client and server, is managed by the [`utils/protocol_spec.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/protocol_spec.js) module. This ensures a consistent and validated communication layer across Playwright's architecture.

Additionally, utility scripts handle specific documentation-related tasks. The [`utils/generate_chromium_default_font_families.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_chromium_default_font_families.js) script parses Chromium's resource XML files to extract and transform default font family settings for different operating systems into a structured JSON format. The [`utils/generate_cli_help.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_cli_help.js) script generates CLI help information in various formats, including README, plain text, or JSON. ### Dependency Auditing and Enforcement

The Playwright monorepo maintains stringent controls over its dependencies to ensure stability, security, and architectural coherence. These controls involve enforcing internal dependency rules, validating [`package.json`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/workspace.js#L37) declarations, and performing automated security audits.

Internal dependency rules are established through [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files, which reside within package directories and specify permissible internal module imports. The script [`utils/build_deps_true.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build_deps_true.js) automates the generation of [`DEPS.true`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/validate_deps.js#L21) files. These files analyze TypeScript/JavaScript source files to identify their actual dependencies, recording the resolved import targets while excluding type-only imports and honoring [`@no-check-deps`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js#L190) comments for specific exceptions. This process provides a clear and verifiable record of dependencies, which helps maintain the monorepo's internal structure.

To ensure consistency between declared and actual dependencies, the script [`utils/check_deps.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js) validates [`package.json`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/workspace.js#L37) declarations. It parses source code to identify all [`import`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/LICENSE#L77) statements and compares these external dependencies against those listed in the [`package.json`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/workspace.js#L37) file. The script flags any discrepancies, such as undeclared but used dependencies or version mismatches between core Playwright packages. Furthermore, this tool verifies internal imports against the rules defined in [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files, upholding the monorepo's architectural guidelines. Vendored packages are exempt from these [`package.json`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/workspace.js#L37) checks, as their authorization stems directly from the [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) rules.

Security auditing is integrated into the development workflow via [`utils/check_audit.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_audit.js). This script executes [`npm audit`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L252) to detect potential security vulnerabilities within the project's npm dependencies. It then filters these findings against a list of known, acceptable advisories, allowing the team to focus on new or unaddressed vulnerabilities. If any new vulnerabilities are identified after this filtering process, the audit fails, necessitating immediate attention and remediation to maintain the security posture of the Playwright ecosystem.

### Code Quality and Linting Tools

The Playwright project maintains code quality through several specialized linting tools. One such tool is an ESLint plugin that enforces proper usage of [`progress`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/isomorphic/teleSuiteUpdater.ts#L47) objects in asynchronous functions. This helps in consistently propagating progress tracking throughout asynchronous operations.

Beyond JavaScript, Playwright also employs comprehensive linters for code snippets embedded in its documentation, supporting JavaScript/TypeScript, Python, C#, and Java. The central script for this linting process is [`utils/doclint/linting-code-snippets/cli.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/cli.js). This utility extracts code blocks from Markdown files and dispatches them to language-specific linting services. For JavaScript/TypeScript, it integrates directly with ESLint. For other languages, it leverages external tools: Python snippets are checked against [`black`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/overflow.html#L8) formatting rules via [`utils/doclint/linting-code-snippets/python/main.py`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/python/main.py), C# snippets undergo syntax validation using [`Microsoft.CodeAnalysis`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/csharp/Program.cs#L1) in [`utils/doclint/linting-code-snippets/csharp/Program.cs`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/csharp/Program.cs), and Java snippets are syntax-checked using [`javaparser`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/java/pom.xml#L18) in [`utils/doclint/linting-code-snippets/java/src/main/java/JavaSyntaxChecker.java`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/linting-code-snippets/java/src/main/java/JavaSyntaxChecker.java). These services run external linters in child processes, ensuring consistency and correctness of code examples across various programming languages in the project's documentation.

### Linux Browser Dependency Management

Playwright manages browser dependencies on Linux distributions by identifying required shared libraries and mapping them to corresponding system packages. This ensures that Playwright can provide accurate suggestions for installing missing dependencies when a browser cannot launch due to unmet library requirements.

The process for generating this dependency map is automated through the [`utils/linux-browser-dependencies/run.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/linux-browser-dependencies/run.sh) script. This script orchestrates the creation of a Docker container for a specified Linux distribution. Inside this isolated environment, [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) is installed, and its browser binaries are downloaded.

A key component within the Docker container is the [`utils/linux-browser-dependencies/inside_docker/list_dependencies.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/linux-browser-dependencies/inside_docker/list_dependencies.js) script. This script analyzes the browser executables to detect any missing shared libraries using tools like [`ldd`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/README.md?plain=1#L3). Once missing libraries are identified, it performs a reverse lookup using [`apt-file search`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/linux-browser-dependencies/README.md?plain=1#L28) to determine which system packages provide those libraries. The script also incorporates logic to resolve ambiguities when multiple packages might offer the same library, favoring non-development packages and allowing for interactive selection if necessary. The result is a mapping that links specific shared libraries to their respective installable packages, which is then used by Playwright to guide users in setting up their environments.

### Flakiness Dashboard Reporting and Upload

The Playwright project utilizes an Azure Function-based flakiness dashboard to aggregate and display test reports, providing insights into test stability. This system processes raw test data, compresses it for efficient storage, and makes it available for analysis through a web interface.

The core of this system is the [`utils/flakiness-dashboard`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard) directory, which hosts the Azure Function. The function's main entry point, located in [`utils/flakiness-dashboard/processing/index.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/index.js), orchestrates the ingestion of new test reports. When a report is uploaded, it is initially processed to manage storage and then parsed from a gzipped JSON format. This raw data is then passed to two subsequent processing stages: first to [`processDashboardRaw`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_raw.js#L19) and then to [`processDashboardCompressedV1`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_compressed_v1.js#L19).

The [`processDashboardRaw`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_raw.js#L19) function, defined in [`utils/flakiness-dashboard/processing/dashboard_raw.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_raw.js), is responsible for storing the original test report data. It aggregates all reports associated with a specific Git commit SHA into a single gzipped JSON blob. This ensures that all raw data for a given commit is maintained together, allowing for detailed historical analysis.

Following raw data storage, the [`processDashboardCompressedV1`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_compressed_v1.js#L19) function, found in [`utils/flakiness-dashboard/processing/dashboard_compressed_v1.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/dashboard_compressed_v1.js), transforms the raw reports into a more compact and organized JSON structure. This process involves cleaning metadata, flattening nested test specifications, and standardizing test names. The compressed data is then gzipped and uploaded to a blob storage, optimizing both storage space and retrieval performance for the flakiness dashboard.

Utilities for blob storage interaction and data compression/decompression are provided by [`utils/flakiness-dashboard/processing/utils.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/utils.js). This file includes [`SimpleBlob`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/utils.js#L47) for simplified Azure Blob Storage operations and [`gzipAsync`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/testserver/index.ts#L34) / [`gunzipAsync`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/utils.js#L24) for efficient data handling. A [`flattenSpecs`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/utils.js#L33) utility helps in normalizing test suite structures, which is crucial for consistent data representation in the dashboard. Additionally, [`utils/flakiness-dashboard/processing/compress_raw_data.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/processing/compress_raw_data.js) allows for the reprocessing of existing raw dashboard data for a given Git SHA into the compressed V1 format.

The deployment of this Azure Function is managed through standard Azure tooling, as outlined in [`utils/flakiness-dashboard/README.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/README.md), involving [`Azure Functions Core Tools`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/README.md?plain=1#L3) and [`Azure CLI`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/ci.md?plain=1#L459) for authentication and publishing. The aggregated and processed data is ultimately consumed by [`https://devops.playwright.dev/flakiness.html`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/flakiness-dashboard/README.md?plain=1#L4), providing a comprehensive view of test flakiness trends.

### Interactive Dashboard for Remote Browser Control

The Playwright Dashboard, located in [`packages/dashboard`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/dashboard), provides an interactive web application for controlling and monitoring remote browser sessions. The dashboard's architecture is rooted in a client-server communication model. ### Browser Extension for Existing Page Integration

The Playwright browser extension enables Playwright to interact with existing browser pages, leveraging the user's default authenticated state. This allows for automation tasks on websites where a user is already logged in, bypassing the need for separate authentication within Playwright scripts.

At a high level, the extension functions by establishing a connection between a Playwright client and the browser's debugging interface. This connection is managed through a client-server architecture where the extension acts as a server relaying commands and events. The extension supports different protocol versions, with Protocol V1 handling a single-tab legacy approach and Protocol V2 offering a more flexible, multi-tab reflective mechanism.

Core to the extension's operation is the background script, residing in [`packages/extension/src/background.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/background.ts). This script manages connection requests from external Playwright clients, oversees the lifecycle of debugging sessions, and handles interactions with browser tabs. It coordinates the attachment and detachment of the Playwright debugger to active tabs and ensures proper cleanup of resources, especially in the context of Manifest V3 (MV3) service workers which have ephemeral lifecycles.

Connection management involves [`RelayConnection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/relayConnection.ts#L46) objects (defined in [`packages/extension/src/relayConnection.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/relayConnection.ts)), which establish WebSocket connections to a Playwright relay. These connections forward browser events and debugger commands. The specific handling of these commands and events is abstracted into [`ProtocolHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/types/protocol.d.ts#L14790) implementations (found in [`packages/extension/src/protocolHandlers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/protocolHandlers.ts)), which are chosen based on the protocol version. For instance, [`ProtocolV2Handler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/relayConnection.ts#L26) allows dynamic invocation of [`chrome.*`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/extension/extension-fixtures.ts#L101) methods and direct forwarding of all [`chrome.*`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/extension/extension-fixtures.ts#L101) events, providing a rich interaction model.

The user interface (UI) components, located in [`packages/extension/src/ui`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui), facilitate user interaction with the extension. The [`ConnectApp`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/connect.tsx#L30) (in [`packages/extension/src/ui/connect.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/connect.tsx)) manages the process of connecting a Playwright client to a browser tab, including parsing connection parameters, validating URLs, and handling protocol version compatibility. Users can select specific tabs for interaction. For automated scenarios, an authentication token mechanism is available. The [`AuthTokenSection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/authToken.tsx#L22) (in [`packages/extension/src/ui/authToken.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/authToken.tsx)) allows users to generate, store, and display a secure token that can bypass manual connection approval, enabling headless or automated environments to seamlessly connect. The [`StatusApp`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/status.tsx#L22) (in [`packages/extension/src/ui/status.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/status.tsx)) provides feedback on active client connections and allows users to navigate to connected tabs. Visual consistency across the UI is maintained through a centralized styling system, including theming support for light and dark modes, defined in files like [`packages/extension/src/ui/colors.css`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/colors.css) and [`packages/extension/src/ui/connect.css`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/extension/src/ui/connect.css).

The extension's design emphasizes the ability to seamlessly integrate with a user's existing browser context, leveraging current authentication and enabling direct interaction with web pages without requiring a separate browser launch or login process from Playwright. This approach is particularly useful for AI assistants (LLMs) or complex automation scenarios where maintaining session state is crucial.

### Recorder for Code Generation and Inspection

The Playwright Recorder provides a frontend application for capturing browser interactions, generating Playwright test code, and debugging test steps. It serves as the user interface for the Playwright Inspector, facilitating the creation and management of automated tests.

The Recorder's primary interface is rendered by the [`Recorder`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/library/inspector/inspectorTest.ts#L108) React component, defined in [`packages/recorder/src/recorder.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorder.tsx). This component orchestrates the various UI elements for user interaction. The application is initiated by [`packages/recorder/src/index.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/index.tsx), which sets up the basic HTML structure in [`packages/recorder/index.html`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/index.html) and applies a theme.

Core functionalities include:

- **Recording User Interactions**: Users can start and stop recording their actions within a browser. These interactions, such as clicks, fills, and navigations, are captured and translated into actionable Playwright code.
- **Code Generation and Display**: As interactions are recorded, the Recorder generates Playwright test code in real-time, which is displayed in a dedicated [`CodeMirrorWrapper`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/web/src/components/codeMirrorWrapper.tsx#L52) component within the UI. This allows developers to immediately see the code corresponding to their actions. The [`packages/recorder/src/actions.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/actions.d.ts) file defines the types for these various recorder actions (e.g., [`ClickAction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/actions.d.ts#L49), [`FillAction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/actions.d.ts#L70)) and signals (e.g., [`NavigationSignal`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/actions.d.ts#L139)), providing a structured representation of user flows.
- **Element Inspection and Locator Picking**: The Recorder includes tools to inspect elements on a web page and generate appropriate locators for them. This simplifies the process of identifying and targeting specific UI components in tests.
- **Assertion Generation**: The tool supports generating assertions for properties like visibility, text content, element values, and visual snapshots, enhancing the reliability of generated tests. An [`autoExpect`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorder.tsx#L48) setting can automate the creation of these assertions.
- **Call Log Viewing**: A [`CallLogView`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/callLog.tsx#L30) component, styled by [`packages/recorder/src/callLog.css`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/callLog.css) and implemented in [`packages/recorder/src/callLog.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/callLog.tsx), displays a chronological log of recorded actions. This log provides details on each step, including its status and duration, aiding in debugging and review.
- **Playback Controls**: The Recorder offers controls to [`resume`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/clock.ts#L212), [`pause`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/page.ts#L841), and [`step`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L90) through recorded actions, mimicking debugger-like functionality. These controls allow developers to examine the execution of their recorded steps in detail.
- **Settings Management**: A [`Dialog`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/recorder/recorder.ts#L1720) component enables users to configure various settings, such as the application's theme and whether [`autoExpect`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorder.tsx#L48) is enabled.

The communication between the frontend Recorder and its backend is managed through [`window.sendCommand`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorder.tsx#L280) for sending commands and [`window.dispatch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorder.tsx#L98) for receiving updates. The [`packages/recorder/src/recorderTypes.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorderTypes.d.ts) file defines the TypeScript interfaces for this communication, including [`RecorderBackend`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorderTypes.d.ts#L108) methods called by the frontend and [`RecorderFrontend`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/src/recorderTypes.d.ts#L119) methods called by the backend to update the UI state. The build process for the Recorder is configured using [`vite`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L159) via [`packages/recorder/vite.config.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/recorder/vite.config.ts), which sets up React plugin integration and defines path aliases for shared modules.

### Trace Viewer for Post-Mortem Analysis

The Playwright Trace Viewer provides a web-based interface for analyzing Playwright traces, enabling post-mortem debugging and understanding of test execution. It visualizes recorded actions, console output, network requests, and errors, offering a comprehensive overview of browser interactions during a test run. The viewer is built using React and relies on a service worker to efficiently load and serve trace data.

The main entry point for the Trace Viewer is [`packages/trace-viewer/index.html`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/index.html), which loads the core application logic from [`packages/trace-viewer/src/index.tsx`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/index.tsx). The service worker, defined in [`packages/trace-viewer/src/sw/main.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/sw/main.ts), plays a crucial role in intercepting network requests and serving trace data, snapshots, and resources directly from loaded trace files. This design allows for performant and reliable trace viewing, whether the trace is a local ZIP file or streamed live.

A central feature of the Trace Viewer is its ability to visualize snapshots, displaying web page states at different points in a trace. This is handled by [`packages/trace-viewer/snapshot.html`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/snapshot.html), which embeds an [`iframe`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/protocol.d.ts#L5190) to securely display page content, with its functionality managed by [`packages/trace-viewer/src/snapshotMain.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/snapshotMain.ts). The UI components for the viewer, located in [`packages/trace-viewer/src/ui`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/ui), provide interactive elements for:

- **Action Lists**: Displaying a filterable and interactive list of traced actions, showing their status, duration, and method details.

## Component Testing

Playwright's component testing capabilities provide a way to test UI components in isolation within a browser environment. This experimental functionality, encapsulated primarily within the [`packages/playwright-ct-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core) package, integrates with Playwright's test runner to offer features for component lifecycle management, interaction, and configuration. The package does not adhere to semantic versioning, indicating that its API and behavior may evolve.

Component testing is configured using the [`defineConfig`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/test.mjs#L32) function, which extends Playwright's test configuration with component-specific options. These options, defined in [`packages/playwright-ct-core/index.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/index.d.ts), include settings for the component testing server port, template directory, cache directory, and custom Vite configurations. This allows developers to tailor the build process for components, supporting various frameworks like React and Vue.

The core of component interaction lies in the [`mount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_3.1.5.js#L5328) fixture, provided through [`packages/playwright-ct-core/src/mount.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts). This fixture allows components to be rendered into the browser's DOM, managed via [`innerMount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L107), [`unmount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_3.1.5.js#L3048), and [`update`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_2.6.14.js#L4552) functions. These functions interact with global browser-side methods like [`window.playwrightMount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L111) and [`window.playwrightUpdate`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L103), which are responsible for the actual rendering and updates. Communication between the browser and the Node.js test runner is handled through [`page.exposeFunction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/api/class-page.md?plain=1#L1862) for callbacks via [`__ctDispatchFunction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L51). Components are consistently mounted into a [`div`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/frames/style.css#L1) element with [`id='root'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/html-reporter/index.html#L26).

For handling dynamic component imports and inter-process communication between the test runner and the browser, a set of utilities are injected into the browser environment. These utilities, found in [`packages/playwright-ct-core/src/injected`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected), include an [`ImportRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L27) instance available as [`window.__pwRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/index.ts#L20) for managing and resolving dynamic component imports. Serialization and deserialization of objects, functions, and import references are handled by [`window.__pwUnwrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L101) and [`window.__pwTransformObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-react/registerSource.mjs#L48) (from [`packages/playwright-ct-core/src/injected/serializers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts)), which enable robust data exchange.

The build process for components leverages Vite. The [`packages/playwright-ct-core/src/vitePlugin.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/vitePlugin.ts) file provides a [`TestRunnerPlugin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/plugins/index.ts#L20) that orchestrates the Vite build and serving. This plugin handles the creation of the Vite bundle, starting a preview server, and tracking dependencies. The [`packages/playwright-ct-core/src/viteUtils.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/viteUtils.ts) file supports this integration by managing Vite configuration, resolving directories, and transforming the entry file for component registration. A custom Vite plugin is injected during the build to record file timestamps and collect component dependencies, ensuring efficient and incremental builds.

Network request routing and interception during component tests are managed by the [`Router`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L85) class, defined in [`packages/playwright-ct-core/src/router.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts). This class allows for the registration of [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) instances to intercept and handle network requests, providing fine-grained control over the network environment. It also overrides [`globalThis.fetch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L28) to ensure consistent request handling and integrates with tools like MSW by injecting cookies and managing passthrough requests.

To support different web frameworks, a Babel plugin ([`packages/playwright-ct-core/src/tsxTransform.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts)) transforms TypeScript JSX (TSX) code. This plugin rewrites imports for non-JavaScript assets and JSX-referenced components into an internal reference format, which the [`ImportRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L27) can then resolve at runtime. This ensures that Playwright can correctly load and test components written in various JSX-based frameworks.

### Core Architecture for Component Testing

Playwright's component testing (CT) functionality is provided by the [`playwright-ct-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L28) package, which is designed to integrate UI component testing directly into the Playwright test runner. This package is marked as experimental and does not adhere to semantic versioning, indicating that its API and behavior may evolve without strict backward compatibility guarantees. For official documentation, users are directed to Playwright's dedicated component testing guides.

The core design of [`playwright-ct-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L28) involves extending the Playwright test runner to enable component-specific interactions. This extension is achieved through the definition of a specialized [`test`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/mcp/fixtures.ts#L86) object that incorporates fixtures tailored for component management. A key aspect of this design is the [`defineConfig`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/test.mjs#L32) function, which serves as the primary mechanism for configuring the component testing environment. This function wraps Playwright's standard configuration to automatically inject necessary plugins and transformations. For instance, it integrates with Vite for efficient component bundling and serving, and includes Babel transformations for processing TypeScript JSX (TSX) files, allowing for dynamic compilation during tests.

This architecture enables developers to test UI components in an isolated, browser-like environment, leveraging Playwright's robust browser automation capabilities. The underlying framework handles the complexities of setting up a component development server, injecting necessary browser utilities, and facilitating communication between the Node.js test runner and the browser environment where components are rendered.

Playwright's component testing capabilities manage the lifecycle of UI components within the browser's Document Object Model (DOM), encompassing their initial rendering, subsequent updates, and eventual removal. This process is orchestrated through specialized test fixtures that facilitate interaction between the Node.js test runner and the browser environment where components are executed.

The core mechanism for bringing a component into the browser's DOM is the [`mount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_3.1.5.js#L5328) fixture, which is extended into Playwright's test runner via [`packages/playwright-ct-core/index.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/index.js). This fixture leverages [`innerMount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L107) in [`packages/playwright-ct-core/src/mount.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts) to asynchronously handle the rendering process. Components are consistently mounted within a [`div`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/frames/style.css#L1) element identified by [`id='root'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/html-reporter/index.html#L26), simplifying their location and manipulation during testing.

Once a component is mounted, its state or properties can be dynamically altered using the [`update`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_2.6.14.js#L4552) method, which internally calls [`innerUpdate`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L97) in [`packages/playwright-ct-core/src/mount.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts). This allows tests to simulate user interactions or data changes and verify the component's reactive behavior. When a component is no longer needed, the [`unmount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_3.1.5.js#L3048) method removes it from the DOM.

A critical aspect of this lifecycle management is the communication channel between the Node.js test runner and the browser. This is established through [`page.exposeFunction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/api/class-page.md?plain=1#L1862) in Playwright, which registers [`__ctDispatchFunction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L51) as a global function within the browser. This function, along with [`boundCallbacksForMount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/mount.ts#L27), enables callbacks from the browser back to the Node.js environment, allowing the test runner to receive information or trigger actions based on browser-side events.

For components that include functions or complex objects in their properties, a serialization and deserialization mechanism is employed. [`wrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L28) in [`packages/playwright-ct-core/src/injected/serializers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts) transforms functions into serializable [`FunctionRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L19) objects with an [`ordinal`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/to-do-notifications/scripts/todo.js#L349) identifier before they are sent to the browser. Conversely, [`unwrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L42) in [`packages/playwright-ct-core/src/injected/serializers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts) reconstructs these references back into callable functions within the browser environment. This ensures that component properties, including callbacks, can be seamlessly transferred and utilized across the Node.js and browser contexts.

The type definitions for these component interactions, including [`JsxComponent`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L17), [`ObjectComponentOptions`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L27), and the extension of the global [`Window`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/baseTest.ts#L47) interface with Playwright-specific functions like [`playwrightMount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L41), [`playwrightUnmount`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L42), and [`playwrightUpdate`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts#L43), are outlined in [`packages/playwright-ct-core/types/component.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/types/component.d.ts). This provides a clear contract for how components are represented and how their lifecycle is managed within Playwright's component testing framework.

### Injected Browser Utilities and Object Serialization

Within the Playwright Component Testing framework, a set of utilities is injected directly into the browser environment to facilitate the dynamic management of components and robust inter-process communication between the test runner and the browser. These injected utilities primarily focus on an [`ImportRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L27) for handling dynamic component imports and specialized functions for serializing and deserializing JavaScript objects, particularly functions and import references.

The [`ImportRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L27), defined in [`packages/playwright-ct-core/src/injected/importRegistry.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts), serves as a central mechanism for registering and resolving component imports at runtime. It maintains a mapping of string identifiers to functions that return Promises resolving to component modules. This allows components to be loaded on demand using a unique [`ImportRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L17) object, which specifies the component's identifier and an optional property to access within the imported module. The registry's ability to [`resolveImportRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L35) ensures that components can be dynamically instantiated and utilized within the browser's DOM, enabling flexible test scenarios for various component frameworks.

For seamless communication between the Node.js test runner and the browser, functions like [`unwrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L42) and [`transformObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L55), implemented in [`packages/playwright-ct-core/src/injected/serializers.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts), manage the serialization and deserialization of complex JavaScript objects. When objects, especially those containing functions, are passed from Node.js to the browser, [`transformObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L55) replaces the actual function implementations with [`FunctionRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L19) objects, which are lightweight references. Conversely, [`unwrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L42) takes these serialized objects and reconstructs them in the browser. It replaces [`FunctionRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L19) objects with proxy functions that, when invoked, dispatch the call back to the Node.js environment via [`window.__ctDispatchFunction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L46). This mechanism ensures that callbacks and interactive functionalities within components can be properly executed and managed across the process boundary. Additionally, [`unwrapObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/serializers.ts#L42) integrates with the [`ImportRegistry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L27) to resolve [`ImportRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/importRegistry.ts#L17) objects back into actual component modules, completing the cycle of dynamic loading and interaction. These core utilities, exposed globally on the [`window`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/baseTest.ts#L47) object by [`packages/playwright-ct-core/src/injected/index.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/injected/index.ts), establish a foundation for interactive and dynamically loaded component testing in Playwright.

### Vite Integration for Component Bundling

Playwright's component testing integrates Vite for bundling and serving components, optimizing the testing workflow for modern web applications. The core integration is managed by a [`TestRunnerPlugin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/plugins/index.ts#L20) defined in [`packages/playwright-ct-core/src/vitePlugin.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/vitePlugin.ts), which orchestrates the build process and serves components using Vite's development server capabilities.

The [`TestRunnerPlugin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/plugins/index.ts#L20) hooks into the Playwright test lifecycle, managing the setup and teardown of the Vite environment. During the [`begin`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/plugins/index.ts#L25) phase of a test run, it triggers [`buildBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/vitePlugin.ts#L105) to compile the components. This function is responsible for determining the framework-specific configuration and resolving output directories, as handled by utilities in [`packages/playwright-ct-core/src/viteUtils.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/viteUtils.ts). It intelligently checks for existing builds and source file changes to perform incremental builds, avoiding unnecessary recompilations.

A custom Vite plugin, also defined in [`packages/playwright-ct-core/src/vitePlugin.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/vitePlugin.ts), is injected into the Vite build process. This plugin records file timestamps and transforms the HTML entry point to inject component registration code. Critically, it employs [`collectViteModuleDependencies`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/vitePlugin.ts#L272) to recursively gather all direct and transitive dependencies of each component. This dependency information is then used by Playwright to track changes and trigger re-runs when a component or its dependencies are modified.

After a successful build, the plugin starts a Vite [`preview`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/javascript.ts#L230) server to serve the bundled components. The base URL of this server is then configured for Playwright's test environment, ensuring that tests interact with the correctly served components. This approach centralizes the build and serving logic within Vite, streamlining the component testing setup. The [`createConfig`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/viteUtils.ts#L72) function in [`packages/playwright-ct-core/src/viteUtils.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/viteUtils.ts) generates a comprehensive Vite configuration, merging base settings with user-defined and framework-specific overrides, including heuristics for JSX support and build optimizations.

### Network Request Routing and Interception

Network request routing and interception within Playwright's component testing framework allow developers to control how components interact with network resources. The central mechanism for this is the [`Router`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L85) class, defined in [`packages/playwright-ct-core/src/router.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts). This class manages network requests within a given [`BrowserContext`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js#L896), enabling the registration of [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) instances.

The [`Router`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L85) class provides a [`use`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/reading-list/vue_3.1.5.js#L5272) method to register [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) instances, allowing multiple handlers to process requests in a chained fashion. Each [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) implements a [`run`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/build.js#L100) method to process an incoming [`Request`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L86) and optionally provide a [`Response`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L648). The [`Router`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L85) intercepts all network requests using a catch-all route ([`'**/*'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/codegen.md?plain=1#L454)). When a request is intercepted, it is converted into a standard Web API [`Request`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L86) object and passed to the registered [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) s. If a handler returns a [`response`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L648), that response is used to fulfill the intercepted request.

A key design aspect of the [`Router`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L85) is its ability to override [`globalThis.fetch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L28) when request handlers are active. This ensures that [`fetch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/fetch.ts#L170) calls made directly by the client-side code are also routed through the [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) chain, providing consistent interception capabilities. This override specifically handles compatibility with tools like Mock Service Worker (MSW), checking for headers such as [`x-msw-intention`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L41) to identify requests that MSW intends to bypass. For such bypassed requests, the [`globalFetch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L50) mechanism injects cookies from the Playwright [`BrowserContext`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js#L896), maintaining session consistency. The system uses a counter to manage [`fetch`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/fetch.ts#L170) overrides across multiple [`BrowserContext`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/firefox/juggler/TargetRegistry.js#L896) instances, ensuring that only the active context's cookies are used for [`msw/passthrough`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L45) requests.

The type definitions for these capabilities, including the [`RouterFixture`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/index.d.ts#L41) and [`RequestHandler`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/router.ts#L19) interface, are provided in [`packages/playwright-ct-core/index.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/index.d.ts), integrating these features seamlessly into the Playwright test API.

### TSX Transformation and Import Handling

The Playwright component testing framework includes a Babel plugin, defined in [`packages/playwright-ct-core/src/tsxTransform.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts), to process TypeScript JSX (TSX) code. This plugin is essential for handling how components and their dependencies are imported and referenced within the testing environment.

The plugin operates by visiting different nodes in the Abstract Syntax Tree (AST) of the TSX files. When it encounters [`ImportDeclaration`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L74) nodes, it identifies imports that are either "artifact extensions" (such as [`.vue`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/test-components-js.md?plain=1#L152), image files, CSS files, or font files, defined in [`artifactExtensions`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L178)) or JavaScript modules used directly as JSX components. These imports are transformed into an internal [`importRef`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L62) format, which is a standardized way for Playwright to resolve and load these assets or components dynamically at runtime. This transformation involves replacing the original import statements and generating unique identifiers for each transformed import.

Additionally, the plugin analyzes [`MemberExpression`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L113) nodes to ensure that references to these transformed imports are correctly adjusted. Before performing these transformations, the plugin uses a preliminary pass to identify all local names within the code that are used as JSX component tags via [`collectJsxComponentUsages`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L132). This pre-analysis allows the [`ImportDeclaration`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-ct-core/src/tsxTransform.ts#L74) visitor to make informed decisions about which JavaScript imports require transformation because they are explicitly used as JSX components. After processing, the plugin injects new [`const`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L229) declarations into the file, representing the transformed imports as runtime objects that the Playwright component testing framework can understand and utilize. This ensures that the component testing environment can correctly resolve and render both traditional JavaScript/TypeScript components and external assets.

## Development and DevOps Workflows

This section outlines the development and operational workflows for the Playwright project, encompassing API development, tool creation, dependency management, and DevOps practices for continuous integration (CI) analysis.

Development within Playwright involves a structured approach to extending its capabilities. New APIs are defined first in documentation, which then drives the auto-generation of public and test API types. Client-side logic for interacting with the backend is implemented using [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32), with parameters grouped and timeouts managed for robust communication. The Remote Procedure Call (RPC) protocol is formally defined in `/microsoft/playwright/packages/protocol/src/protocol.yml`, leading to the auto-generation of TypeScript interfaces, runtime validators, and method metadata. Dispatchers route these RPC calls to server objects, which contain the core logic for browser interaction. This server logic often delegates to browser-specific implementations for Chromium ([`packages/playwright-core/src/server/chromium/crPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/chromium/crPage.ts)), Firefox ([`packages/playwright-core/src/server/firefox/ffPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/firefox/ffPage.ts)), and WebKit ([`packages/playwright-core/src/server/webkit/wkPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/wkPage.ts)), ensuring compatibility across different browsers. For a detailed guide on this process, see [Playwright API Development Workflow](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-playwright-api-development-workflow).

The project adheres to a client-server architecture where the client exposes the public API and the server handles browser automation. Communication between these layers is governed by an RPC protocol. This architecture is enforced by strict dependency rules specified in [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files within each directory. These rules prevent client code from importing server code directly and vice-versa, ensuring that all cross-layer communication occurs solely through the defined protocol. This design promotes modularity and maintainability. Further information on the architectural layers and dependency management can be found in [Client-Server Architecture and Dependency Management](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-client-server-architecture-and-dependency-management).

Playwright also supports the development of internal tools and Command-Line Interface (CLI) commands, referred to as MCP tools. These tools are defined with schemas and handlers, registered centrally in [`packages/playwright-core/src/tools/backend/tools.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tools.ts), and can be integrated into the CLI. New configuration options can be added by modifying type definitions, CLI parsing logic, and environmental variable handling across several core files, such as [`packages/playwright-core/src/tools/mcp/config.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.d.ts) and [`packages/playwright-core/src/tools/mcp/program.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/program.ts). This extensibility allows for the creation of specialized utilities that interact deeply with Playwright's browser automation capabilities. For details on building such tools, refer to [Developing MCP Tools and CLI Commands](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-developing-mcp-tools-and-cli-commands).

Vendor dependencies and internal code are managed through a bundling strategy that inlines third-party npm packages and Playwright's own modules into optimized bundle files, such as `/microsoft/playwright/lib/utilsBundle.js` and `/microsoft/playwright/lib/coreBundle.js`. This process involves mapping npm imports to bundle exports, ensuring that dependencies are correctly resolved and bundled for distribution. The [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files also play a role here, authorizing specific [`node_modules`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L27) imports. This bundling strategy is critical for managing the project's external dependencies and ensuring efficient package delivery, as described in [Vendor Dependency and Bundling Strategy](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-vendor-dependency-and-bundling-strategy). For example, updating the [`BROWSER_VERSION`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/wkBrowser.ts#L35) in [`packages/playwright-core/src/server/webkit/wkBrowser.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/wkBrowser.ts) requires running [`npm run flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L3) to update generated files. When reproducing regressions, especially across published versions, developers are guided to work directly with installed [`node_modules`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L27) content rather than the monorepo source to identify the root cause and verify fixes by in-place patching.

The project also maintains a comprehensive tracing framework, allowing for the detailed recording and visualization of automation runs. This system captures various events, including API calls, network traffic in HAR format, DOM snapshots, and screencast frames, all correlated by a unique [`callId`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/utils.ts#L129) for each action. Traces are stored in a structured directory and can be loaded and viewed using a dedicated Trace Viewer application that provides a graphical interface for post-mortem analysis. The viewer supports lazy loading of data and handles older trace formats through a modernizer. For an in-depth understanding of this system, see [Playwright Trace System and Viewer](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-playwright-trace-system-and-viewer).

In terms of DevOps, Playwright includes capabilities for analyzing CI failures and debugging workflows. A key component is the generation of a [`main`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L121) branch failure report, which categorizes failures into regressions, pre-existing flakiness, or infrastructure issues. This report provides a detailed breakdown of failing tests and error messages, aiding in rapid debugging and issue resolution. A script named [`.claude/skills/playwright-devops/fetch-commit-logs.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-devops/fetch-commit-logs.sh) automates the process of downloading failed job logs from GitHub Actions, centralizing the necessary data for analysis. This structured approach to CI failure analysis helps maintain the stability and quality of the Playwright codebase. More information on this can be found in [CI Failure Analysis and Reporting](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-ci-failure-analysis-and-reporting) and [Automated Log Fetching for CI Workflows](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-automated-log-fetching-for-ci-workflows).

### Playwright API Development Workflow

The process of extending Playwright's API involves a structured, multi-layered approach, beginning with documentation and extending through client and server implementations. API specifications are first defined in Markdown files, which serve as the source for public API types. These documentation files dictate the structure of methods, parameters, and events.

The client-side API uses [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32) as a base class for public API objects. These objects interact with the backend via a [`_channel`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/utils/nodePlatform.ts#L131) proxy that intercepts method calls and sends RPC messages. Parameters are grouped into single objects for channel calls, and channel return values are unwrapped or converted for use.

The API's protocol channel is defined in `/microsoft/playwright/packages/playwright-core/src/protocol/protocol.yml`. This YAML definition specifies methods, parameters, and return types, which are then used to auto-generate TypeScript interfaces, runtime validators, and method metadata. This ensures type safety and consistency across the client-server boundary.

Dispatcher handlers are implemented to bridge [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49) instances to the protocol. Dispatchers receive validated parameters and a [`Progress`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/protocol/src/progress.d.ts#L33) object, routing calls to corresponding server objects. They convert dispatcher references to actual server objects and wrap server objects in results back into dispatchers.

The core server-side logic resides in files like [`packages/playwright-core/src/server/page.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/page.ts), where methods implement the actual browser interaction. These methods often delegate to browser-specific implementations, such as those found in [`packages/playwright-core/src/server/chromium/crPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/chromium/crPage.ts), [`packages/playwright-core/src/server/firefox/ffPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/firefox/ffPage.ts), and [`packages/playwright-core/src/server/webkit/wkPage.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/webkit/wkPage.ts), which use their respective browser protocols. For a deeper understanding of the client-server architecture, refer to [Client-Server Architecture and Dependency Management](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-client-server-architecture-and-dependency-management).

### Client-Server Architecture and Dependency Management

Playwright's client-server architecture enables communication between the client-facing API and the browser automation backend. This separation ensures that the public API remains consistent while allowing for browser-specific implementations on the server side. The Remote Procedure Call (RPC) protocol, defined by the [`protocol.yml`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L9) file, forms the backbone of this communication. This file dictates the structure of commands, events, and data types exchanged between the client and server, with code generation tools producing TypeScript interfaces and runtime validators from this definition.

The client layer, primarily composed of [`ChannelOwner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/channelOwner.ts#L32) classes, provides the public API objects such as [`Page`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/page.ts#L164), [`Frame`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/frame.ts#L48), and [`Browser`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/browser.ts#L70). These objects manage their RPC connections and act as proxies, intercepting method calls and translating them into RPC messages for the server. The [`Connection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/connection.ts#L69) class handles the underlying client-server transport, managing the lifecycle of remote objects and dispatching incoming messages and events. Event subscriptions are optimized so that messages are only sent to the client when active listeners are present.

On the server side, [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49) serves as the base for all domain objects, carrying information like a unique identifier and an attribution chain for tracing. Browser-specific operations are often delegated through a [`PageDelegate`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/page.ts#L57) pattern, with distinct implementations for Chromium, Firefox, and WebKit leveraging their native browser protocols.

Bridging the gap between the server-side [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49) instances and the RPC protocol are the [`Dispatcher`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/runner/dispatcher.ts#L35) classes. Each dispatcher wraps an [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49), exposing methods that align with the defined protocol channels. Dispatchers are responsible for converting incoming RPC parameters into server objects and wrapping server results back into dispatchers for client consumption. The [`DispatcherConnection`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/dispatchers/dispatcher.ts#L192) manages these dispatchers on the server, validating messages, invoking instrumentation hooks, and overseeing object lifecycle events, including garbage collection.

Internal dependency management within the monorepo is governed by [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files located in each directory, which enforce strict import rules. The [`flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L52) command-line tool verifies these rules, ensuring that modules adhere to their declared dependencies. For instance, client-side code is explicitly prevented from directly importing server-side code, and vice versa, reinforcing the RPC-based communication model. Vendored npm packages are also managed and bundled, with specific mapping configurations ensuring they are correctly integrated and externalized where necessary. More details on Playwright's development workflow can be found in [Playwright API Development Workflow](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-playwright-api-development-workflow).

### Developing MCP Tools and CLI Commands

New backend tools and command-line interface (CLI) commands in Playwright are developed by defining MCP (Microsoft Playwright) tools with schemas and handlers, registering them, and declaring CLI commands using [`declareCommand`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-daemon/command.ts#L36). Extending configuration options involves modifications across several files to ensure proper type definition, CLI parsing, environment variable handling, and merging.

Backend tools are defined in files within the `/microsoft/playwright/packages/playwright-core/src/tools/backend/` directory. These tools are created using either [`defineTool`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L58) or [`defineTabTool`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L70). [`defineTabTool`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L70) is typically used for operations tied to a specific browser tab and automatically manages modal states, while [`defineTool`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L58) provides access to the full browser context, suitable for broader operations not confined to a single tab. Each tool requires a [`schema`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/types/protocol.d.ts#L22784) to define its inputs, name, title, description, and type (e.g., [`'action'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/utils.ts#L126), [`'input'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/utils.ts#L130), [`'readOnly'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/injected/src/injectedScript.ts#L761), [`'assertion'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/mcp/test/testBackend.ts#L29)). The core logic of the tool resides in an asynchronous [`handle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L619) function, which interacts with the Playwright browser context or tab to perform its designated tasks. Communication back to the client is managed through a [`Response`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L648) API within the [`handle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/network.ts#L619) function, allowing for sending text, errors, code, snapshots, and images. Tools are associated with [`ToolCapability`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L33) (e.g., [`'core'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L33), [`'storage'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L33), [`'devtools'`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/program.ts#L94)), which determines when they are enabled, and must be registered in the [`browserTools`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tools.ts#L47) array within [`packages/playwright-core/src/tools/backend/tools.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tools.ts).

CLI commands act as wrappers for these MCP tools. They are declared in [`packages/playwright-core/src/tools/cli-daemon/commands.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-daemon/commands.ts) using [`declareCommand`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-daemon/command.ts#L36). A command definition includes its name, description, category, and specifications for positional [`args`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/documentation.js#L350) and named [`options`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/test-cli-js.md?plain=1#L178). It maps these CLI parameters to the [`toolName`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/response.ts#L56) and [`toolParams`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-daemon/daemon.ts#L94) of an underlying MCP tool, enabling users to interact with backend functionalities directly from the command line. Command categories, such as those defined in [`packages/playwright-core/src/tools/cli-daemon/command.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-daemon/command.ts), help organize CLI functionalities and require corresponding entries for help generation.

Extending Playwright's configuration options involves modifying [`Config`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/types/test.d.ts#L2045) and [`ToolCapability`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/backend/tool.ts#L33) in [`packages/playwright-core/src/tools/mcp/config.d.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.d.ts) for type definitions. The [`CLIOptions`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts#L35) in [`packages/playwright-core/src/tools/mcp/config.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts) must also be updated. Configuration resolution, including mapping CLI options and environment variables, is managed within [`configFromCLIOptions`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts#L274) and [`configFromEnv`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts#L370) in [`packages/playwright-core/src/tools/mcp/config.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts). CLI flags are added in [`packages/playwright-core/src/tools/mcp/program.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/program.ts), and [`mergeConfig`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts#L452) in [`packages/playwright-core/src/tools/mcp/config.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/mcp/config.ts) handles the merging logic for nested options, ensuring a consistent configuration precedence (default → config file → environment variables → CLI arguments).

For a deeper dive into how Playwright's client-server architecture and dependency management operate, refer to [Client-Server Architecture and Dependency Management](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-client-server-architecture-and-dependency-management). Additionally, the overall Playwright development workflow, including API development and vendor dependency management, is detailed in [Playwright API Development Workflow](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-playwright-api-development-workflow).

### Vendor Dependency and Bundling Strategy

Playwright manages its external npm dependencies and internal code through a bundling strategy that integrates these components into pre-built files, minimizing the footprint of [`node_modules`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L27) and streamlining deployment.

Playwright utilizes several bundles:

- [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) bundles:
	- [`packages/playwright-core/src/utilsBundle.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/utilsBundle.ts) contains vendored npm packages that are essential for Playwright's runtime operations, such as [`debug`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/reporters/github.ts#L48) and [`ws`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/stress/static/js/main.cc4a7955.js#L2).
		- The core bundle re-exports modules from [`playwright-core`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/README.md?plain=1#L1) itself, consolidating client, isomorphic, and utility code.
- [`playwright`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/cli.js#L20) bundles:
	- The transform bundle wraps Babel-related packages, providing a shared resource for code transformation needs.
		- Other bundles, such as the common and runner bundles, serve as aggregators for related modules, managing singletons and externalizing shared dependencies like the transform bundle to prevent duplication.

Code that is not specifically included in a bundle entry is compiled individually by [`esbuild`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/generate_injected.js#L23) and placed into the `/microsoft/playwright/lib/` directory, preserving its original source layout.

Each bundle generates sidecar files for transparency and licensing:

- [`.js.txt`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/bundle_report.js#L96) files provide human-readable reports detailing inlined files, external dependencies, and size information.
- [`.js.LICENSE`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/bundle_report.js#L139) files contain third-party license texts for all included npm packages, which are then aggregated into top-level [`ThirdPartyNotices.txt`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/.npmignore#L44) files.

The process for adding a new npm dependency involves several steps to ensure it is correctly integrated into the [`utilsBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js#L101): installing the package, exporting it from [`packages/playwright-core/src/utilsBundle.ts`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/utilsBundle.ts), mapping it in [`utils/build/utilsBundleMapping.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/utilsBundleMapping.js), and updating the [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) for dependency enforcement. The [`npm run flint`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L3) command is used to execute checks and ensure proper integration.

Internal third-party helpers, not managed via npm, reside in `/microsoft/playwright/packages/playwright-core/src/server/utils/third_party/`. These are either re-exported via `/microsoft/playwright/packages/playwright-core/src/server/utils/index.ts` and accessed using the [`@utils/*`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tsconfig.json#L18) path alias, or, if they import npm packages, those packages are still routed through the [`utilsBundle`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js#L101).

Dependency rules are enforced by [`DEPS.list`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L92) files located in each directory under `/microsoft/playwright/packages/`. These files control permissible imports, distinguishing between internal source files, aliased paths, and [`node_modules`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/package.json#L27) packages. The [`utils/check_deps.js`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/check_deps.js) script verifies these rules, ensuring that modules only import allowed dependencies. Further details on dependency management can be found in [Client-Server Architecture and Dependency Management](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-client-server-architecture-and-dependency-management).

### Playwright Trace System and Viewer

Playwright's trace system provides a comprehensive framework for recording and visualizing browser interactions. It captures a wide array of data points including API calls, network requests, DOM snapshots, video frames, console messages, and errors, all correlated to provide a complete picture of an automation run for post-mortem analysis and debugging.

The system defines various trace event types, such as [`ContextCreatedTraceEvent`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/trace.ts#L34), [`BeforeActionTraceEvent`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/trace.ts#L62), [`InputActionTraceEvent`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/trace.ts#L78), and [`AfterActionTraceEvent`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/trace.ts#L98) in [`.claude/skills/playwright-dev/trace_system_guide.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-dev/trace_system_guide.md), which structure the recorded data. Network traffic is captured using types based on the HTTP Archive (HAR) format, including [`HARFile`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/har.ts#L18), [`Log`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/trace-viewer.md?plain=1#L735), and [`Entry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/utils/third_party/yauzl/index.d.ts#L14) in [`.claude/skills/playwright-dev/trace_system_guide.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-dev/trace_system_guide.md), with additional custom fields for Playwright-specific data. DOM snapshots, crucial for visualizing the state of the web page, are defined by [`FrameSnapshot`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/snapshot.ts#L40) and [`NodeSnapshot`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/snapshot.ts#L29) in [`.claude/skills/playwright-dev/trace_system_guide.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-dev/trace_system_guide.md), enabling compact encoding of the DOM tree and embedding of resource data via [`ResourceOverride`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace/src/snapshot.ts#L34). These definitions are central to the integrity and interpretability of the trace data.

The [`Tracing`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/client/tracing.ts#L26) class, an [`SdkObject`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/instrumentation.ts#L49), is responsible for orchestrating data collection. It records action metadata, stack traces, DOM snapshots ([`onBeforeCall`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/recorder.ts#L406), [`onBeforeInputAction`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/recorder.ts#L428), [`onAfterCall`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/recorder.ts#L418)), API logs ([`onCallLog`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/server/recorder.ts#L459)), network requests, console messages ([`onConsoleMessage`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/api/class-page.md?plain=1#L188)), browser events, screencast frames, and process output. A key design pattern is the use of a unique [`callId`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/config/utils.ts#L129) to link all related events for a single action, ensuring comprehensive correlation of data points. The system also records [`wallTime`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/traceModel.ts#L73) and [`monotonicTime`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/time.ts#L35) for precise timing and supports chunked recording, allowing traces to be saved in multiple segments.

Trace data is stored in a structured directory, typically containing [`.trace`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/html-reporter/src/links.css#L77) files (JSONL for events), [`.network`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/ui/networkTab.css#L17) files (JSONL for network events), and a [`resources/`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/traceLoader.ts#L117) directory for binary blobs, indexed by SHA1 hash.

Trace loading is handled by the [`TraceLoaderBackend`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/traceLoader.ts#L24) interface, which defines methods for reading trace files. A built-in [`ZipTraceLoaderBackend`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/sw/traceLoaderBackends.ts#L26) in [`.claude/skills/playwright-dev/trace_system_guide.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-dev/trace_system_guide.md) is used for loading [`.zip`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock.md?plain=1#L419) files. The loading process involves parsing trace and network files, modernizing events, and building a [`ContextEntry`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/entries.ts#L23) array that contains pages, resources, actions, events, stdio, and errors. The [`TraceModernizer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/traceModernizer.ts#L38) class ensures backward compatibility by upgrading older trace formats to the current version.

The [`TraceModel`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/isomorphic/trace/traceModel.ts#L66) class provides a high-level data model for the trace viewer. It aggregates and organizes all loaded trace data, including pages, actions, attachments, events, stdio, errors, resources, and sources. This model facilitates the rendering of various aspects of the trace data in the UI.

The Trace Viewer is a UI application that visualizes the recorded traces. It utilizes a [`Service Worker`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/network.md?plain=1#L768) in [`packages/trace-viewer/src/sw`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/sw) to intercept trace URL fetches. The viewer incorporates lazy loading for snapshots and resources, which are stored by SHA1 and fetched on demand, and uses snapshot references to compactly encode DOM changes. The [`show-trace`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L100) command allows users to launch the trace viewer, either in a browser tab or a dedicated application window. Further details on the Trace Viewer's architecture can be found in [Trace Viewer for Post-Mortem Analysis](https://codewiki.google/github.com/microsoft/playwright#playwright-ecosystem-and-extensions-trace-viewer-for-post-mortem-analysis).

### CI Failure Analysis and Reporting

The Playwright project incorporates a robust DevOps process for analyzing Continuous Integration (CI) failures, particularly focusing on generating a comprehensive health report for the [`main`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L121) branch. This process aims to identify, categorize, and debug issues that arise during automated testing.

At the core of this system is the capability to generate a CI failure report, as outlined in [`.claude/skills/playwright-devops/commit-failures.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-devops/commit-failures.md). This report is designed to provide a holistic view of CI stability by analyzing the most recent commit on the [`main`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L121) branch. Rather than simply indicating whether a commit passed or failed, it categorizes failures into "Possible regressions" (issues likely introduced by the commit), "Pre-existing / flaky" (intermittent failures, timeouts, or infrastructure-related problems), and "Infrastructure issues" (non-test failures). This categorization helps developers understand the root cause of failures more effectively.

The generation of this report involves two phases. The initial phase, "Fetch logs," is orchestrated by the [`bash`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/README.md?plain=1#L29) script at [`.claude/skills/playwright-devops/fetch-commit-logs.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-devops/fetch-commit-logs.sh). This script automates the downloading of failed job logs from GitHub Actions. It utilizes the [`gh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/react/react-dom@17.0.2.production.min.js#L81) command-line tool to interact with the GitHub API and [`jq`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-client/skill/SKILL.md?plain=1#L189) for JSON processing, creating a structured temporary directory containing commit information, details of failed workflows, metadata for failed jobs, and the full log output for each individual failed job. The script employs parallel execution for faster log retrieval and ensures that logs from both failed and in-progress workflows are captured. The [`sanitize`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/generateApiJson.js#L109) function within this script is used to clean up string inputs for use as directory or file names, ensuring validity. The [`REPO`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/api-testing-js.md?plain=1#L73) variable defines the target GitHub repository for fetching logs.

The subsequent "Analyze" phase processes the collected log data. This involves reading the summary file, generated by the log-fetching script, to retrieve commit details and lists of failed workflows and jobs. Each [`.log`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/trace-viewer/src/ui/logTab.css#L17) file is then parsed to extract failing test names and their associated error messages. The final report is compiled into a Markdown format, offering a summary section that provides an overview of failures and a detailed section that organizes granular information by workflow and job, presenting each failing test with its error message in a tabular format. This structured approach facilitates quick identification and debugging of issues within the Playwright CI pipeline. Further details on the overall DevOps capabilities are available in [Development and DevOps Workflows](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows).

### Automated Log Fetching for CI Workflows

The [`.claude/skills/playwright-devops/fetch-commit-logs.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-devops/fetch-commit-logs.sh) script automates the retrieval of failed job logs from GitHub Actions for a given Git commit. This script is a component in the process of [CI Failure Analysis and Reporting](https://codewiki.google/github.com/microsoft/playwright#development-and-devops-workflows-ci-failure-analysis-and-reporting) within the Playwright project.

Its primary function is to download and organize logs, enabling developers to diagnose issues. The script utilizes the [`gh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/react/react-dom@17.0.2.production.min.js#L81) command-line tool to interact with the GitHub API and [`jq`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-client/skill/SKILL.md?plain=1#L189) for JSON processing. It can be executed with a specific commit SHA or will default to fetching logs for the latest commit on the [`main`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/browser_patches/winldd/PrintDeps.cpp#L121) branch.

A key design choice of [`.claude/skills/playwright-devops/fetch-commit-logs.sh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/.claude/skills/playwright-devops/fetch-commit-logs.sh) is its ability to fetch logs for individual jobs in parallel, which reduces the time required for data collection. This parallel execution is managed by spawning multiple [`gh run view`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/CLAUDE.md?plain=1#L43) commands concurrently. To ensure proper organization and file system compatibility, the script includes a [`sanitize`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/doclint/generateApiJson.js#L109) function that cleans up workflow and job names for use as directory and file names.

The output of the script is structured within a temporary directory, containing individual log files for each failed job. Additionally, it generates a [`SUMMARY`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/bundle_report.js#L169) JSON object that provides a structured overview of the commit, including details of failed workflows and their associated jobs. This [`SUMMARY`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/utils/build/bundle_report.js#L169) is used for the subsequent analysis phase where failures are categorized and reported. The reliance on [`gh`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/tests/assets/react/react-dom@17.0.2.production.min.js#L81) and [`jq`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright-core/src/tools/cli-client/skill/SKILL.md?plain=1#L189) streamlines interaction with GitHub's API, ensuring efficient and robust log retrieval.

### API Testing with GitHub

The [`examples/github-api`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/github-api) directory provides an example of using Playwright for automated API testing against a RESTful API, specifically GitHub's API. This approach demonstrates how to manage API requests, handle authentication, and assert response validity within a test framework.

This comprehensive example demonstrates a structured approach to API testing, covering authentication, request configuration, and assertion strategies.

The core of this testing strategy involves injecting a script into the browser context *before* the application under test loads. This is achieved by methods within the mock object, such as [`_setLevel`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/tests/update-battery-status.spec.js#L27) and [`_setCharging`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/tests/update-battery-status.spec.js#L31), which allow programmatic alteration of the battery's [`level`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/tests/update-battery-status.spec.js#L14) and [`charging`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/tests/update-battery-status.spec.js#L15) status. When these methods are invoked, they also trigger [`chargingchange`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock-browser-js.md?plain=1#L112) and [`levelchange`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock-browser-js.md?plain=1#L113) events, simulating real browser behavior. This allows tests to verify that the application's user interface updates correctly in response to these simulated changes, such as displaying a new percentage or indicating a change in power source.

Additionally, the testing framework provides mechanisms to verify that the web application correctly interacts with the Battery Status API. This includes logging calls to [`getBattery`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock-browser-js.md?plain=1#L12) and [`addEventListener`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/utils/eventsHelper.ts#L31) for [`chargingchange`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock-browser-js.md?plain=1#L112) and [`levelchange`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/docs/src/mock-browser-js.md?plain=1#L113) events. By analyzing these logs, testers can confirm that the application fetches battery information and registers for relevant events as expected, without making actual hardware calls. This controlled environment ensures that tests are isolated, reproducible, and do not depend on the physical state of the testing machine. The example implementation can be found in [`examples/mock-battery/demo-battery-api`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/demo-battery-api) and its corresponding tests in [`examples/mock-battery/tests`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/mock-battery/tests).

### AI-Driven Test Automation for TodoMVC

The TodoMVC example showcases an architecture for AI-driven test automation, orchestrated to manage the entire lifecycle of Playwright tests, from initial planning to generation, and subsequent self-healing. This system is defined within the [`examples/todomvc`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc) directory, leveraging a series of AI agents to achieve comprehensive test coverage and maintainability.

- **Test Planner**: The [`playwright-test-planner`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/agents/generateAgents.ts#L298) agent, as described in [`examples/todomvc/.claude/agents/playwright-test-planner.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc/.claude/agents/playwright-test-planner.md), creates detailed test plans for web applications. - **Test Generator**: The [`playwright-test-generator`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/agents/generateAgents.ts#L299) agent, defined in [`examples/todomvc/.claude/agents/playwright-test-generator.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc/.claude/agents/playwright-test-generator.md), takes these test plans and translates them into executable Playwright test code. It simulates user interactions within the browser and captures these actions to produce well-structured test files.
- **Test Healer**: The [`playwright-test-healer`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/packages/playwright/src/agents/generateAgents.ts#L300) agent, detailed in [`examples/todomvc/.claude/agents/playwright-test-healer.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc/.claude/agents/playwright-test-healer.md), acts as a debugging and self-correction mechanism. It identifies failing tests, investigates errors by leveraging browser and test tools, performs root cause analysis, and remediates test code to resolve issues.

The overall workflow is orchestrated by a central script, [`examples/todomvc/.claude/prompts/playwright-test-coverage.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc/.claude/prompts/playwright-test-coverage.md) (and [`examples/todomvc/.github/prompts/playwright-test-coverage.prompt.md`](https://github.com/microsoft/playwright/blob/c1b1aa6bcc3566962b1b53b2e4ea9f68704c1ea4/examples/todomvc/.github/prompts/playwright-test-coverage.prompt.md)). This script coordinates the planner, generator, and healer agents. It first invokes the planner to create a test plan, then iterates through each test case in the plan to generate individual Playwright tests, and finally engages the healer to run these tests and fix any failures. This ensures that the test suite remains robust and up-to-date with application changes.
