	 # Notes on Playwright Assertions

## Overview
- Playwright uses the **expect** API for validating application behavior in UI and API tests.[page:1][web:1]  
- There are two main categories: **web‑first locator assertions** that auto‑retry, and **generic assertions** for plain values that do not retry.[web:1][page:2]  
- Choosing locator assertions for UI checks greatly reduces flakiness by leveraging Playwright’s built‑in auto‑waiting.[web:1][page:2]  

---

## Assertion Types in Playwright

### 1. Locator (Web‑First) Assertions
- Locator assertions work on `Locator`, `Page`, and `APIResponse` objects and automatically retry until the condition is met or the timeout elapses.[web:1][web:12]  
- Example: `await expect(page.getByTestId('status')).toHaveText('Submitted');` re‑queries the DOM until the element’s text becomes `"Submitted"` or the timeout is hit.[web:1][web:12]  
- Default assertion timeout is 5 seconds and can be customized per assertion or globally via `testConfig.expect.timeout`.[web:1]  

**Common locator assertions**[web:1]:  
- `toBeVisible()`, `toBeHidden()`, `toBeInViewport()` – visibility and viewport checks.  
- `toBeEnabled()`, `toBeDisabled()`, `toBeEditable()` – control interaction state.  
- `toContainText()`, `toHaveText()` – text content validation.  
- `toHaveAttribute()`, `toHaveClass()`, `toHaveCSS()`, `toHaveId()` – DOM attributes and CSS.  
- `toHaveCount()` – number of matching elements.  
- `toHaveValue()`, `toHaveValues()`, `toBeEmpty()` – form value checks.  
- `toHaveRole()`, `toHaveAccessibleName()`, `toHaveAccessibleDescription()` – accessibility.  
- `toHaveScreenshot()` on `page` or `locator` – visual comparisons.[web:1]  
- `toHaveTitle()`, `toHaveURL()` – page metadata and navigation checks.[web:1]  
- `toBeOK()` on `APIResponse` – asserts that an API response has an OK (2xx) status.[web:9][web:12]  

### 2. Generic Assertions
- Generic assertions operate on plain values and do a single evaluation without auto‑retry.[web:1][page:2]  
- They are typically used for data and computation checks rather than DOM state.[page:2][web:6]  

**Examples of generic matchers**[web:1]:  
- Equality and structure: `toBe()`, `toEqual()`, `toStrictEqual()`, `toMatchObject()`.  
- Collections and strings: `toContain()`, `toContainEqual()`, `toHaveLength()`, `toMatch()` (regex).  
- Type/value checks: `toBeDefined()`, `toBeUndefined()`, `toBeNull()`, `toBeNaN()`, `toBeTruthy()`, `toBeFalsy()`, `toBeInstanceOf()`.  
- Numbers: `toBeGreaterThan()`, `toBeGreaterThanOrEqual()`, `toBeLessThan()`, `toBeLessThanOrEqual()`, `toBeCloseTo()`.  
- Errors: `toThrow()` for functions that should error.  

**Key distinctions (generic vs locator)**[page:2]:  
- Generic: no `await`, argument is a plain value (e.g., `const result = 2 + 2; expect(result).toBe(4);`).  
- Locator: always `await`, argument is a locator or page object (e.g., `await expect(page.locator('#submit-button')).toBeVisible();`).

## Negation, Asymmetric Matchers, and Soft Assertions

### Negating matchers
- Any matcher can be inverted by chaining `.not`, e.g. `expect(value).not.toEqual(0)` or `await expect(locator).not.toContainText('error');`.[web:1]  

### Asymmetric / flexible matching
Used inside other assertions to relax exact matching rules.[web:1]  

- `expect.any(Type)` – accepts any instance of a given class/primitive.  
- `expect.anything()` – accepts any non‑null/undefined value.  
- `expect.arrayContaining([...])` – array contains specific elements.  
- `expect.arrayOf(Type)` – all array items are of a given type.  
- `expect.objectContaining({ ... })` – object contains specified properties.  
- `expect.stringContaining(substr)` – string contains substring.  
- `expect.stringMatching(regex)` – string matches regex.  

### Soft assertions
- Soft assertions mark the test as failed but **do not** stop execution.[web:1]  
- Syntax: `await expect.soft(page.getByTestId('status')).toHaveText('Success');`.[web:1]  
- You can later assert that no soft assertion failed: `expect(test.info().errors).toHaveLength(0);`.[web:1]  
- Soft assertions only work when using the Playwright Test runner.[web:1]

## Configuring and Extending `expect`

### Custom messages
- `expect` accepts an optional second argument as a custom message, improving failure diagnostics.[web:1]  
- Example: `await expect(page.getByText('Name'), 'should be logged in').toBeVisible();` shows this message for both pass and fail in reports.[web:1]  

### `expect.configure`
- You can create preconfigured `expect` instances with custom defaults like `timeout` or `soft`.[web:1]  
- Example for slow checks:  
  ```ts
  const slowExpect = expect.configure({ timeout: 10000 });
  await slowExpect(locator).toHaveText('Submit');
- Example for always‑soft assertions:
  ```
    ts
    
    `const softExpect = expect.configure({ soft: true }); await softExpect(locator).toHaveText('Submit'); ```[web:1]`  
    

## `expect.poll`

- Wraps any function in a polling assertion that retries until the nested generic assertion passes.[web:1]
    
- Common for asynchronous conditions outside the DOM, such as API status eventually becoming 200.[web:1]
    
- Example:
    
    ts
    
    `await expect.poll(   async () => (await page.request.get('https://api.example.com')).status(),  { message: 'make sure API eventually succeeds', timeout: 10000 } ).toBe(200); ```[web:1]`  
    

## `expect.toPass`

- Retries an entire block of assertions until all inside pass or a timeout/interval budget is reached.[web:1]
    
- Default timeout is 0 (effectively single run) unless you pass custom `timeout` and `intervals`.[web:1]
    
- Example:
    
    ts
    
    `await expect(async () => {   const response = await page.request.get('https://api.example.com');  expect(response.status()).toBe(200); }).toPass({ intervals: , timeout: 60000 }); ```[web:1]`  
    

## Custom matchers (`expect.extend`)

- You can add domain‑specific matchers that integrate with Playwright’s reporting.[web:1]
    
- A custom matcher must return `{ pass, message, name, expected, actual }` and may internally reuse other Playwright assertions.[web:1]
    
- Example: `toHaveAmount(locator, expected)` that internally uses `toHaveAttribute('data-amount', String(expected))` to assert an amount on a cart element.[web:1]

## Best Practices for Assertions in Playwright

## Prefer web‑first locator assertions

- Use locator assertions (`await expect(locator)....`) for anything DOM‑related, rather than first reading values and then asserting with generics.[page:2][web:3][web:4]
    
- Example: prefer `await expect(page.locator('#myInput')).toHaveValue('Expected Text');` over reading value then `expect(inputValue).toBe('Expected Text');` to leverage auto‑retry.[page:2]
    

## Keep assertions specific and stable

- Make assertions as **targeted** as possible (e.g., assert exact text on a dedicated test id, not on a generic container) to avoid false positives.[web:3]
    
- Avoid monolithic assertions that cover many independent aspects; instead, use multiple focused `expect` calls, optionally grouped in test steps.[web:3]
    

## Manage timeouts correctly

- Use Playwright’s built‑in assertion timeout instead of hard waits like `waitForTimeout`.[web:1][page:2]
    
- Configure a higher global `expect.timeout` for generally slow apps, and override per‑assertion only when necessary:
    
    ts
    
    `await expect(page.locator('#slow-element')).toBeVisible({ timeout: 30000 }); ```[page:2]`  
    

## Hard vs soft assertions

- Treat the default (hard) assertions as gatekeepers for critical preconditions (e.g., login success, required elements present).[web:1][web:5]
    
- Use soft assertions when you want to collect multiple failures in a single test (e.g., validating many form fields) without aborting early.[web:1][web:5]
    

## Visual and URL/title assertions

- Use `toHaveURL()` and `toHaveTitle()` as navigation guards to ensure the app reaches the correct pages after actions.[web:1][web:5]
    
- For visual regressions, `toHaveScreenshot()` (page or locator) compares pixel by pixel against a stored baseline; a mismatched region surfaces clearly in the diff image.[web:1][web:5]
    

## Avoid common pitfalls

- Do not mix Jest’s standalone `expect` library with Playwright’s `expect`, since only the latter is integrated with Playwright Test’s auto‑waiting, timeouts, and reporting.[web:1][web:12]
    
- Avoid manual polling loops or `while`/`setTimeout` for conditions that can be expressed with locator assertions, `expect.poll`, or `expect.toPass()`.[web:1][web:3]

## Practical Assertion Patterns

## Visibility and presence

- Visible/hidden (UI state):
    
    ts
    
    `await expect(page.locator('#display-text')).toBeVisible(); await expect(page.locator('#display-text')).toBeHidden();        // or: .not.toBeVisible() ```[web:1][web:5]`  
    
- Present/not present (DOM presence via count):
    
    ts
    
    `await expect(page.locator('.list-item')).toHaveCount(0);         // not present await expect(page.locator('.list-item')).toHaveCount(3);         // three elements present ```[page:2]`  
    

## Enabled/disabled

- Check interaction states:
    
    ts
    
    `await expect(page.getByRole('button', { name: 'Go to Home' })).toBeEnabled(); await expect(page.getByRole('button', { name: 'Disabled Button' })).toBeDisabled(); ```[web:1][web:5]`  
    

## Text match / mismatch

- Exact text:
    
    ts
    
    `await expect(page.locator('#submit-button')).toHaveText('Submit');`
    
- Partial or negative:
    
    ts
    
    `await expect(page.locator('#alert')).toContainText('Success'); await expect(page.locator('#alert')).not.toContainText('Error'); ```[web:1][web:5]`  
    

## Attribute assertions

- DOM attributes like `name`, `placeholder`, `class`:
    
    ts
    
    `await expect(page.locator('input[name="username"]'))   .toHaveAttribute('placeholder', 'Username'); await expect(page.locator('input[name="username"]'))   .toHaveAttribute('class', /oxd-input/); ```[web:1][web:5]`  
    

## URL and title

- Full and partial URL:
    
    ts
    
    `await expect(page).toHaveURL('https://demo.orangehrmlive.com/'); await expect(page).toHaveURL(/demo\.orangehrm/i); ```[web:1][web:5]`  
    
- Title:
    
    ts
    
    `await expect(page).toHaveTitle('OrangeHRM'); await expect(page).toHaveTitle(/HRM$/); ```[web:1][web:5]`  
    

## Screenshot assertions

- First run establishes a baseline; subsequent runs compare pixel by pixel.[web:1][web:5]
    
- Example:
    
    ts
    
    `await expect(page).toHaveScreenshot(); await expect(page.locator('.hero-section')).toHaveScreenshot('hero.png'); ```[web:1][web:5]`

## Conceptual Role of Assertions (High‑Level)

- Assertions ensure that automated tests truly validate application behavior rather than just executing steps.[web:5]
    
- They support validation, early error detection, and overall automation quality by enforcing checks at key points in the flow (e.g., after login, after navigation, after submission).[web:5]
    
- In Playwright specifically, rich assertion types (visibility, state, text, attributes, URL/title, screenshots, API responses) make it possible to cover both functional and visual aspects of web apps effectively.[web:1][web:5]
